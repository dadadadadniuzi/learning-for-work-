# exec函数族

## 本节学什么

- `execl`
- `execlp`
- `execv`
- `execvp`
- 参数传递方式与 PATH 搜索




## 本节学什么详解

- `execl`：`execl` 解决的是“怎样用参数列表形式，把当前进程替换成指定路径的新程序”这个问题。它适合参数个数比较明确的场景，是理解 `exec` 家族最直观的入口。
- `execlp`：`execlp` 解决的是“怎样像命令行那样，按 `PATH` 搜索程序并用参数列表启动它”这个问题。它和 `execl` 的区别主要在 `p`，正好对应这一节里“PATH 搜索”的主题。
- `execv`：`execv` 解决的是“怎样用参数数组形式，把当前进程替换成指定路径的新程序”这个问题。它适合参数已经组织成数组的情况，和 `execl` 一起帮助你理解 `l` / `v` 的区别。
- `execvp`：`execvp` 解决的是“怎样同时拥有参数数组和 `PATH` 搜索能力”这个问题。它在真实程序里很常见，因为很多时候参数本来就是数组形式。
- 参数传递方式与 PATH 搜索：这一节要分清两件事：参数到底是数组还是列表传入，以及带 `p` 的 `exec` 接口为什么能按 `PATH` 去搜索命令。

## 知识点补充
- `exec` 的本质是用新程序替换当前进程映像。
- `l` 和 `v` 区分参数组织方式，`p` 版本会搜索 `PATH`。
- 成功不会返回，失败才返回，所以失败路径要单独处理。
- exec函数族一般规律：
	exec函数一旦调用成功，即执行新的程序，不返回。只有失败才返回，错误值-1，所以通常我们直接在exec函数调用后直接调用perror()，和exit()，无需if判断。
	l(list) 命令行参数列表
	p(path) 搜索file时使用path变量
	v(vector) 使用命令行参数数组
	e(environment) 使用环境变量数组，不适用进程原有的环境变量，设置新加载程序运行的环境变量
	事实上，只有execve是真正的系统调用，其他5个函数最终都调用execve，是库函数，所以execve在man手册第二节，其它函数在man手册第3节。

## 细节补充

- `wait` 是基础回收，`waitpid` 是更灵活的指定回收
- [[linux系统编程/概念词条/waitpid选项|WNOHANG]] 是非阻塞回收里最常见的选项
- `exec` 成功后不会返回，所以成功路径和失败路径要分开写
- `execl/execlp` 和 `execv/execvp` 的差别在于参数组织方式
- `fork + exec` 是创建新程序的标准套路
