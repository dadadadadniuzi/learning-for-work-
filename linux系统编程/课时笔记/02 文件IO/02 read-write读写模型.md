# read-write读写模型

## 本节学什么

- `read`
- `write`
- 短读短写
- 错误处理与循环读写




## 本节学什么详解

- `read`：`read` 解决的是“怎样从文件描述符对应的数据源里取出字节到用户缓冲区”这个问题。它通常接在 `open` 之后使用，是文件拷贝、输入读取、管道接收等场景的基础。
- `write`：`write` 解决的是“怎样把用户缓冲区里的字节送到文件描述符对应目标里”这个问题。它和 `read` 常成对出现，构成最基本的“读一段、写一段”循环模型。
- 短读短写：短读短写表示系统调用一次返回的字节数可能少于你请求的数量，写循环和读循环就是用来把这种“没读完/没写完”的情况补齐。
- 错误处理与循环读写：文件 I/O 不能只看成功路径，必须处理返回值、`errno` 和短读短写，循环读写是写健壮程序的基本功。

## 知识点补充
- `read` 和 `write` 都是按字节工作的，不保证一次完成。
- 返回值比“是否成功”更重要，要学会处理短读和短写。
- 缓冲区、EOF、循环读写是这一节的核心。

## 本节内容速览

- 文件 IO 不保证一次就完整读完或写完
- 返回值比函数名更重要
- 复制文件的基本模型就是 `read -> write` 循环



## 细节补充

- `open` 的第二个参数既决定访问方式，也决定是否创建、截断、追加
- `read/write` 不保证一次完整完成，返回值是你处理 IO 的核心依据
- `lseek` 只移动偏移，不会改变文件内容本身
- `stat/lstat` 的差别要和软链接联动理解
- `dup/dup2` 的本质是让多个描述符共享同一个打开文件对象
- `fcntl` 是后面非阻塞、描述符控制的常见入口
