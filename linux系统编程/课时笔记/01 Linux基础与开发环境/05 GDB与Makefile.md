# GDB与Makefile

## 本节学什么

- `gdb` 调试流程
- 断点、单步、查看变量、回溯栈
- `Makefile` 的目标、依赖和规则




## 本节学什么详解

- `gdb` 调试流程：调试的顺序通常是：带调试信息编译、进入 `gdb`、设置断点、运行程序、单步执行、查看变量和栈帧、继续运行，最后定位问题。
- 断点、单步、查看变量、回溯栈：这几项是 `gdb` 最核心的定位手段：断点帮你停住，单步帮你看执行路径，变量查看帮你核对数据，回溯栈帮你找调用链。
- `Makefile` 的目标、依赖和规则：Makefile 的本质是描述“做什么、依赖什么、怎么做”：目标是结果，依赖是前提，规则是命令；理解这三项后，你才能读懂自动编译过程。

## 知识点补充
- `gdb` 的价值在于定位程序具体卡在哪一行、哪一个栈帧。
- `break`、`run`、`next`、`step`、`backtrace` 是调试最常用的命令链。
- `Makefile` 用目标、依赖和命令把重复编译自动化。

## 内容速览

- `gcc -g` 让程序带上调试信息，`gdb` 才能对应源码行号和变量。
- 调试时最重要的是知道程序停在哪一行、当前变量是什么、调用栈怎么走过来的。
- `Makefile` 的作用是把重复编译自动化，不是替代编译器。

## GDB 调试流程

1. 用 `gcc -g` 编译程序。
2. 用 `gdb app` 打开可执行文件。
3. 用 `break` 在关键位置打断点。
4. 用 `run` 开始运行。
5. 用 `next`、`step` 单步执行。
6. 用 `print`、`info locals` 查看变量。
7. 用 `backtrace` 查看调用栈。
8. 修改源码后重新编译，再次调试。

## GDB 常用命令

- `run`：运行程序
- `break`：设置断点
- `next`：下一步，不进入函数
- `step`：下一步，进入函数
- `continue`：继续执行到下一个断点
- `print`：打印变量值
- `display`：持续显示变量
- `backtrace`：查看调用栈
- `frame`：切换栈帧
- `list`：查看源码
- `info locals`：查看局部变量
- `quit`：退出调试

## GDB 重点理解

- 没有 `-g`，GDB 很难准确对应源码和变量。
- `next` 和 `step` 的区别在于是否进入函数。
- `backtrace` 解决的是“我现在是怎么走到这里的”。
- `frame` 解决的是“我想看哪一层函数的变量”。

## Makefile 的核心

### 1. 目标

- 目标是要生成什么文件。
- 例如：可执行文件、目标文件、清理动作。

### 2. 依赖

- 目标依赖哪些文件。
- 只要依赖文件变了，目标就需要重新生成。

### 3. 命令

- 满足依赖关系后执行的命令。
- 命令前面必须是 `Tab`，不是空格。

## Makefile 规则格式

```makefile
target: dep1 dep2
	command
```

## 自动变量

- `$@`：目标名
- `$<`：第一个依赖
- `$^`：所有依赖

## 伪目标

- `clean` 常常是伪目标。
- 它不是要生成一个叫 `clean` 的文件，而是执行清理动作。

## 模式规则

- 用一个规则匹配一类文件。
- 作用是减少重复编写。
