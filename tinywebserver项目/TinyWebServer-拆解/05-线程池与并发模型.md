# 线程池与并发模型

返回：[[TinyWebServer-面试拆解笔记]]

相关：[[TinyWebServer-拆解/03-入口与WebServer总控]]、[[TinyWebServer-拆解/04-http_conn与HTTP状态机]]、[[TinyWebServer-拆解/06-定时器与连接管理]]、[[TinyWebServer-拆解/12-半同步半反应堆与并发模型详解]]

## 为什么需要线程池

- 避免每个请求都临时创建线程
- 控制线程数量，防止上下文切换太多
- 主线程专注事件分发，工作线程专注任务处理

## 线程池里有什么

- 工作线程数组
- 任务队列
- 互斥锁
- 信号量
- 数据库连接池指针
- 并发模型标志

## 工作线程怎么运行

工作线程会一直循环：

1. 等待任务
2. 被唤醒后取任务
3. 执行任务
4. 回到等待状态

所以面试如果问“工作线程是一直等待吗”，答案是：是的，它们常驻在线程池里。

## 关键函数

### 构造函数

- 作用：创建固定数量的线程
- 每个线程创建后会 `detach`

### `append(T *request, int state)`

- 作用：把任务放入队列
- 常用于 Reactor 模式
- `state` 表示当前是读任务还是写任务

### `append_p(T *request)`

- 作用：把任务放入队列
- 常用于 Proactor 模式

### `worker(void *arg)`

- 作用：线程入口函数

### `run()`

- 作用：工作线程不断取任务并执行

## Reactor 和 Proactor 在这个项目里怎么体现

### Reactor

- 主线程只负责发现某个 fd 可读或可写
- 把任务交给线程池
- 工作线程真正去做 `read_once()` 或 `write()`

### Proactor

- 主线程先把数据读到用户态缓冲区
- 再把处理任务交给线程池
- 工作线程主要做业务逻辑处理

## 三种模型怎么区分

### Reactor

- 内核通知“可以读写了”
- 应用程序自己完成实际读写

### Proactor

- 理想定义里是内核帮你完成异步 I/O
- 完成后再通知应用

### 主从 Reactor

- 主 Reactor 负责接收新连接
- 从 Reactor 负责处理已连接 socket 的读写事件

## 这个项目整体属于什么模型

整体上，它属于“半同步 / 半反应堆”模型：

- 主线程负责事件监听和分发
- 工作线程同步处理任务

## 如果 1000 个客户端同时访问，线程不多怎么办

重点在于：

- 连接数不等于线程数
- `epoll` 只关注活跃连接
- 线程池只处理当前就绪任务
- 大量连接其实大多数时间都在等待 I/O

所以不是每个连接都需要一个线程。

## 如果一个请求特别耗时，会不会影响其他请求

会。  
因为线程池里的一个线程长时间被占用，其他任务就只能等待。

优化思路：

- 重任务异步化
- 业务线程池拆分
- 设置超时和熔断
- 做缓存
- 把大任务拆成小任务

## 面试里可以怎么说

> 线程池的作用是把线程创建成本前移，避免为每个请求单独建线程，同时把主线程和工作线程的职责分开。这个项目里，主线程负责 epoll 事件监听和任务分发，工作线程负责执行连接处理逻辑。项目同时支持 Reactor 和 Proactor 两种处理思路，区别主要在于谁来完成真正的读写 I/O。
