# TinyWebServer 10分钟总复盘版

> [!abstract]
> 这篇是最后冲刺版。目标不是展开细节，而是让你在 10 分钟内把项目的主干逻辑、核心亮点和高频问题全部过一遍。

## 1. 30秒项目介绍

> 这是一个基于 Linux 高并发网络模型实现的轻量级 C++ Web 服务器。主线程使用 `epoll` 监听连接和 I/O 事件，工作线程通过线程池处理请求；每个客户端连接抽象成 `http_conn` 对象；HTTP 报文解析采用状态机；数据库访问通过 MySQL 连接池；空闲连接通过定时器回收；日志支持同步和异步两种模式；并发模型支持 Reactor 和 Proactor 两种处理思路。

## 2. 项目到底实现了什么

- 静态网页访问
- 图片和视频资源访问
- 用户注册
- 用户登录
- MySQL 数据库交互
- 定时器超时关闭连接
- 同步 / 异步日志
- Reactor / Proactor 模式切换

## 3. 整体架构一口气讲清楚

核心模块有 6 个：

- `WebServer`
  服务器总控，主线程、epoll、监听 socket、信号和定时器调度都在这里
- `http_conn`
  单个 HTTP 连接的读、解析、业务处理和响应构造
- `threadpool`
  工作线程池和任务队列
- `sql_connection_pool`
  MySQL 连接池
- `timer`
  超时定时器链表
- `log`
  同步 / 异步日志

一句话概括结构：

> 主线程负责事件监听和分发，工作线程负责具体处理，数据库连接和日志写入都做了资源复用或解耦。

## 4. 一次请求怎么走

1. 浏览器发起 HTTP 请求
2. 主线程 `epoll_wait()` 等待事件
3. 如果是新连接，就 `accept`
4. 创建对应的 `http_conn`
5. 给连接绑定定时器
6. 如果连接上有读写事件，就按 Reactor / Proactor 模式分发
7. 线程池线程处理请求
8. `http_conn` 用状态机解析 HTTP 报文
9. `do_request()` 执行业务逻辑
10. `process_write()` 组织响应
11. `writev()` 返回给客户端
12. 长连接保留，否则关闭

## 5. HTTP 报文一定要会讲

请求报文由四部分组成：

1. 请求行
2. 请求头
3. 空行
4. 请求体

### 为什么一定要状态机

因为 TCP 是字节流，不保证一次 `recv()` 就读到完整 HTTP 请求。  
所以会出现：

- 一次只收到半个请求头
- 一次收到多个请求
- 请求头和请求体分多次到达

项目的做法是：

- 从状态机：按 `\r\n` 切出完整行
- 主状态机：判断当前是在解析请求行、请求头还是请求体

## 6. GET 和 POST 怎么讲

### GET

- 主要用于访问静态资源
- 一般没有请求体
- 解析完请求行和请求头后就可以进入业务处理

### POST

- 主要用于登录注册
- 需要结合 `Content-Length` 继续解析请求体
- 请求体里通常是表单数据，例如：

```text
user=alice&passwd=123456
```

## 7. 并发模型一定要会讲

### 这个项目整体是什么模型

是“半同步 / 半反应堆”模型：

- 主线程负责事件监听和分发
- 工作线程同步处理任务

### 为什么叫半反应堆

因为主线程只做：

- 监听
- 判断
- 分发

并不把所有业务逻辑都塞在主线程里。

### 为什么叫半同步

因为工作线程拿到任务后，会在当前线程里按顺序把任务处理完。

## 8. Reactor / Proactor / 主从 Reactor 区别

### Reactor

- 内核通知“事件就绪了”
- 真正的读写由应用线程自己做

### Proactor

- I/O 先完成
- 再通知应用做后续逻辑

### 主从 Reactor

- 主 Reactor 负责接收连接
- 从 Reactor 负责处理已连接 socket 的读写事件

### 这个项目怎么对应

- Reactor：线程池线程做真正的 `read/write`
- Proactor：主线程先把数据读好，再交给线程池做逻辑
- 整体不是严格主从 Reactor，更像半同步 / 半反应堆

## 9. 为什么用 epoll

因为它更适合高并发连接场景：

- 只返回就绪 fd
- 适合大量连接
- 支持 LT 和 ET
- 比 `select` 和 `poll` 更高效

### 其他 I/O 复用方式

- `select`
- `poll`
- `epoll`

### 区别一句话

> `select/poll` 更像“每次把所有人点一遍名”，`epoll` 更像“谁准备好了谁自己举手”。

## 10. 线程池一定要会讲

线程池解决的问题：

- 避免频繁创建销毁线程
- 控制线程数量
- 提高并发处理能力

工作线程的状态：

- 平时阻塞等待任务
- 被唤醒后取任务
- 处理完再回到等待状态

如果并发很多但线程不多，为什么还能扛：

- 因为连接数不等于线程数
- `epoll` 只关注当前活跃连接
- 线程池只处理真正就绪的任务

## 11. 数据库连接池一定要会讲

为什么要用连接池：

- 避免每次请求都重新连 MySQL
- 减少连接建立和销毁开销
- 控制数据库连接数量

工作方式：

- 启动时预创建多个连接
- 请求时借一个
- 用完后归还
- 用信号量表示剩余可用连接数

## 12. 登录注册怎么讲

### 登录

1. 浏览器提交 POST 表单
2. 状态机解析请求体
3. 提取用户名密码
4. 在内存 `map` 中校验
5. 成功返回 `welcome.html`
6. 失败返回 `logError.html`

### 注册

1. 解析用户名密码
2. 先看内存 `map` 是否重名
3. 不重名则执行 SQL `INSERT`
4. 成功后跳转登录页
5. 失败跳转注册失败页

## 13. 定时器怎么讲

为什么要用定时器：

- 清理长时间不活跃连接
- 防止 fd 和内存被无效连接一直占着

工作方式：

- 每个连接绑定一个定时器
- 有活动就刷新超时时间
- `alarm + SIGALRM` 周期性触发检查
- 链表头部最早超时，超时就关闭连接

## 14. 日志系统怎么讲

### 同步日志

- 当前线程直接写文件
- 简单，但会阻塞

### 异步日志

- 业务线程先写入阻塞队列
- 后台线程统一写文件
- 更适合高并发

## 15. 静态文件怎么高效返回

项目使用：

- `mmap`
- `writev`

好处：

- 减少拷贝
- 减少系统调用

所以面试里可以说：

> 静态资源不是简单 `read + write`，而是通过 `mmap + writev` 做更高效的发送。

## 16. Webbench 压测怎么讲

Webbench 是轻量级 HTTP 压力测试工具。

常见命令：

```bash
webbench -c 500 -t 30 http://127.0.0.1:9006/judge.html
```

重点指标：

- QPS
- bytes/sec
- 成功请求数
- 失败请求数
- 并发客户端数

面试里最重要的不是说“我测过”，而是说：

> 我会根据吞吐、失败率和不同配置下的结果去分析瓶颈是在主线程、线程池、数据库还是日志 I/O。

## 17. HTTPS 和 GET/POST 高频一句话

### HTTPS 为什么安全

- 加密传输
- 证书认证
- 完整性校验

### GET 和 POST 区别

- GET 主要取资源，参数常在 URL
- POST 主要提交数据，参数常在请求体
- 这个项目里 GET 主要访问静态资源，POST 主要处理登录注册

## 18. 项目亮点怎么讲

- `epoll + 非阻塞 I/O`
- 线程池
- 状态机解析 HTTP
- MySQL 连接池
- 定时器回收空闲连接
- 同步 / 异步日志
- Reactor / Proactor 两种模型切换

## 19. 项目不足怎么讲

不要说“没有不足”，可以说：

- HTTP 支持不完整
- 登录注册逻辑和网络逻辑耦合较紧
- 密码明文存储不安全
- SQL 拼接有注入风险
- 没有完整 Session 体系
- 更偏教学型，不是工业级生产服务器

## 20. 明天上场前最少记住的 12 句话

1. 这是一个基于 `epoll + 线程池 + 非阻塞 I/O` 的轻量级 Web 服务器。
2. 主线程负责监听和分发，工作线程负责处理请求。
3. 每个连接由 `http_conn` 对象维护。
4. HTTP 解析用状态机，因为 TCP 是字节流。
5. GET 主要请求静态资源，POST 主要处理登录注册。
6. 项目整体是半同步 / 半反应堆模型。
7. Reactor 是应用自己做读写，Proactor 是 I/O 完成后再做逻辑。
8. 线程池用于控制线程数量和提高并发处理能力。
9. MySQL 连接池用于复用数据库连接。
10. 定时器用于回收空闲连接。
11. 日志支持同步和异步两种模式。
12. 静态文件使用 `mmap + writev` 返回。
