# TCP数据包格式

TCP 数据包格式通常指 TCP 报文段的首部字段和数据部分。理解它可以帮助你把 [[linux网络编程/概念词条/TCP三次握手|TCP三次握手]]、[[linux网络编程/概念词条/TCP四次挥手|TCP四次挥手]]、序号、确认号、窗口大小这些概念串起来。

## 整体结构

```text
TCP 报文段 = TCP 首部 + TCP 数据
```

TCP 首部最小 20 字节，如果带选项字段会更长。

## TCP 首部字段

| 字段 | 作用 |
|---|---|
| 源端口 | 发送方应用进程对应的端口。 |
| 目的端口 | 接收方应用进程对应的端口。 |
| 序号 Sequence Number | 当前报文段第一个数据字节的序号。建立连接时也用于同步初始序号。 |
| 确认号 Acknowledgment Number | 期望收到的下一个字节序号，表示此前连续数据已经收到。 |
| 首部长度 Data Offset | 表示 TCP 首部有多长，接收方据此找到数据部分起点。 |
| 标志位 Flags | 控制连接建立、关闭、确认等行为，如 `SYN`、`ACK`、`FIN`、`RST`、`PSH`、`URG`。 |
| 窗口大小 Window | 接收方告诉发送方自己还能接收多少数据，用于 [[linux网络编程/概念词条/TCP滑动窗口|TCP滑动窗口]] 和流量控制。 |
| 校验和 Checksum | 检查 TCP 首部和数据在传输中是否出错。 |
| 紧急指针 Urgent Pointer | 配合 `URG` 标志使用，普通 socket 编程入门阶段很少关注。 |
| 选项 Options | 可选字段，如 MSS、窗口扩大、时间戳、SACK 等。 |
| 数据 Data | 应用层交给 TCP 的字节流内容。 |

## 常见标志位

- `SYN`：请求建立连接，在 [[linux网络编程/概念词条/TCP三次握手|TCP三次握手]] 中使用。
- `ACK`：确认号有效，大多数连接建立后的 TCP 报文都会带 ACK。
- `FIN`：请求关闭发送方向，在 [[linux网络编程/概念词条/TCP四次挥手|TCP四次挥手]] 中使用。
- `RST`：异常复位连接，例如连接不存在、端口未监听或程序异常关闭。
- `PSH`：提示接收方尽快把数据交给应用层。
- `URG`：紧急指针有效，入门阶段了解即可。

## 和三次握手的关系

- 第一次握手：客户端发送带 `SYN` 的报文，并携带初始序号。
- 第二次握手：服务器发送 `SYN + ACK`，确认客户端序号，同时发送自己的初始序号。
- 第三次握手：客户端发送 `ACK`，确认服务器序号。

详细见 [[linux网络编程/概念词条/TCP三次握手|TCP三次握手]]。

## 和四次挥手的关系

- 主动关闭方发送 `FIN`。
- 被动关闭方回复 `ACK`。
- 被动关闭方准备关闭时再发送 `FIN`。
- 主动关闭方回复 `ACK`。

详细见 [[linux网络编程/概念词条/TCP四次挥手|TCP四次挥手]]。

## 和滑动窗口的关系

- `Sequence Number` 用来标识字节流位置。
- `Acknowledgment Number` 用来累计确认已经收到的数据。
- `Window` 用来告诉对方接收窗口大小。

这三个字段共同支撑 [[linux网络编程/概念词条/TCP滑动窗口|TCP滑动窗口]]、可靠传输和流量控制。

## 和 socket 编程的关系

- 应用程序调用 [[linux网络编程/函数笔记/Socket/send|send]] 时，不需要自己填写 TCP 首部，内核会自动封装。
- 应用程序调用 [[linux网络编程/函数笔记/Socket/recv|recv]] 时，通常只拿到 TCP 数据部分，不直接看到 TCP 首部。
- 如果想观察 TCP 首部，可以使用抓包工具，例如 [[linux网络编程/指令查询/命令卡/tcpdump|tcpdump]]。

## 易错点

- TCP “数据包格式”属于协议层，普通 socket 代码不手写 TCP 首部。
- TCP 的序号按字节编号，不按 `send` 调用次数编号。
- `Window` 字段表示接收能力，不是应用层数组长度。
