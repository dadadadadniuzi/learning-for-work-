# LungCancerScreening 10分钟速背版

关联：[[LungCancerScreening_项目拆解_面试笔记]] | [[LCS 00 导航]] | [[LungCancerScreening_面试高频问答清单]] | [[LungCancerScreening_口语化逐文件讲解稿]]

> [!summary]
> 这份笔记只保留面试最该记住的内容。
> 目标不是完整，而是让你 10 分钟内把主线过完。

## 1. 一句话介绍项目

这是一个基于 LUNA16 的 AI 辅助肺癌筛查系统，整体采用三阶段流程：先用 2D UNet 做 CT 切片分割找候选区域，再用 3D CNN 判断候选是不是真结节，最后再对真实结节做良恶性风险评估，并通过 Flask 和 React 提供上传、可视化和历史查询功能。

## 2. 30秒版本

这个项目不是单个模型，而是一个完整系统。输入是 `.mhd/.raw` 格式的 CT，第一阶段做切片级分割，第二阶段做真假结节分类，第三阶段做良恶性分析。训练侧有数据集构建、缓存、数据增强、TensorBoard 和模型保存，推理侧有联合评估脚本、Flask 后端、MySQL 持久化和 React 前端页面。

## 3. 核心主线一定要记住

### 三阶段流程

1. 分割：找哪里可疑
2. 分类：判断是不是真结节
3. 恶性分析：判断风险高不高

### 为什么拆三阶段

1. 比端到端更容易解释
2. 更容易调试
3. 更像真实临床辅助流程

## 4. 数据部分要会讲

### 数据来源

主要是 LUNA16：

1. CT 文件：`.mhd/.raw`
2. `annotations.csv`
3. `candidates.csv`
4. `annotations_with_malignancy.csv`

### 数据处理的关键点

1. 读取 CT 体数据
2. 病人坐标和体素坐标互转
3. 从完整 CT 中裁 2D/3D 样本
4. 做缓存和增强

### 为什么要坐标转换

因为标注是病人坐标 `(x, y, z)`，裁图和模型用的是数组坐标 `(index, row, col)`。

## 5. 第一阶段分割怎么讲

### 模型

1. 用 2D UNet
2. 输入不是单张切片，而是 7 通道切片

### 为什么是 7 通道

当前切片前后各 3 张，加上当前切片本身。这样模型能看到上下文，因为结节是 3D 结构。

### 输出

输出是每张切片上的可疑区域 mask。

### 损失函数

主要是 Dice Loss，而且对假阴性额外加权，说明项目更重视“不要漏检”。

### 分割后做什么

做连通域分析，把一片片 mask 转成一个个候选结节中心。

## 6. 第二阶段真假结节分类怎么讲

### 为什么需要第二阶段

因为分割会引入很多误报，比如血管、噪声或者其他组织。

### 输入

以候选点为中心，从完整 CT 中裁一个固定大小 3D 小块，大致是 `32 x 48 x 48`。

### 模型

用 3D CNN 做二分类：

1. 不是结节
2. 是结节

### 为什么用 3D CNN

真假结节判断依赖空间结构，不只是单张切片纹理。

## 7. 第三阶段良恶性分类怎么讲

### 作用

对确认的结节做风险分析，判断它更偏良性还是恶性。

### 模型

结构上和真假结节分类很像，也是在 3D 候选块上做分类。

### 亮点

这里不是完全从零训练，而是基于前一个 3D 分类模型做微调。

### 为什么微调

1. 能复用已有 3D 特征
2. 对小样本任务更稳定

## 8. 训练过程要会概括

每个任务基本都分两步：

1. 先跑缓存脚本
2. 再启动训练脚本

训练时会做：

1. DataLoader
2. 数据增强
3. TensorBoard 记录
4. 验证集评估
5. 保存 checkpoint 和 best model

## 9. 缓存为什么重要

因为 CT 很大，每次都从完整 CT 里重新裁块非常慢，所以作者用 `diskcache + gzip` 把裁好的样本缓存到磁盘，提升训练效率。

## 10. 联合评估脚本怎么讲

`model_evel.py` 的作用是把三个模型串起来做整条 pipeline 的评估，而不是只看单个模型的分数。

它做的事：

1. 分割
2. 候选提取
3. 真假结节分类
4. 恶性分类
5. 和真实标注做空间匹配
6. 输出混淆矩阵

这说明项目不只是训练模型，而是考虑了系统级评估。

## 11. 后端怎么讲

后端基于 Flask，核心类是 `TumorPredictionSystem`。

它封装了完整推理流程：

1. 接收上传文件
2. 读取 CT
3. 跑三阶段模型
4. 汇总结果
5. 存数据库
6. 返回前端

### 重点接口

1. `/api/upload`：上传并预测
2. `/api/predictions`：历史记录
3. `/api/ct-slice/...`：返回带框的切片 PNG
4. `/api/chat`：基于报告上下文问 AI

## 12. 前端怎么讲

前端基于 React，主要页面有：

1. Dashboard：仪表盘
2. Upload：上传和看结果
3. History：历史记录
4. ModelStatus：模型状态
5. Chat：AI 问答

### 前端亮点

1. 支持 CTViewer 看切片
2. 支持历史报告联动 AI 问答
3. 不只是展示数字，而是做成了可交互系统

## 13. 项目亮点一定要会说

1. 不是单一模型，而是全链路系统
2. 三阶段设计合理，可解释性强
3. 有缓存、TensorBoard、模型保存这些工程化细节
4. 有 Flask 后端和 React 前端，产品完成度高
5. 有 CT 切片可视化和报告问答功能

## 14. 项目不足一定要会说

1. 有硬编码，比如数据库和模型路径
2. 有历史遗留逻辑
3. 有些统计数据是模拟的
4. 有少量命名和编码问题

## 15. 如果面试官问“你怎么优化”

你可以答：

1. 配置化路径、数据库和阈值
2. 清理遗留代码和命名问题
3. 增加自动化测试
4. 用真实监控替代模拟统计
5. 进一步优化 3D 模型和后处理

## 16. 最值得背的 1分钟答案

这是一个基于 LUNA16 的 AI 辅助肺癌筛查系统，整体采用三阶段结构。第一阶段用 2D UNet 在 CT 切片上做分割找出疑似结节区域，第二阶段把候选区域裁成固定大小的 3D 体块，用 3D CNN 判断是否是真结节，第三阶段再对真实结节做良恶性风险评估。工程上不仅有模型训练，还包括缓存、联合评估、Flask 后端、MySQL 持久化和 React 前端展示，所以它更像一个完整系统而不是单纯实验代码。它的亮点是全流程完整、可解释性强，但也有一些硬编码和遗留逻辑，后续可以继续做工程化优化。

## 17. 最值得背的 10秒答案

这是一个三阶段肺结节筛查系统，流程是“分割找候选区域、3D 分类确认结节、良恶性风险评估”，并且已经做成了一个带 Flask 后端和 React 前端的可视化系统。
