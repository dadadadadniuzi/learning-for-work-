# 数据库连接池与登录注册

返回：[[TinyWebServer-面试拆解笔记]]

相关：[[TinyWebServer-拆解/04-http_conn与HTTP状态机]]、[[TinyWebServer-拆解/08-日志系统与同步原语]]、[[TinyWebServer-面试问答模拟版]]

## 为什么要数据库连接池

如果没有连接池，每次注册或登录都要：

- `mysql_init`
- `mysql_real_connect`

这样开销很大，而且高并发下连接数也不容易控制。

连接池的思路是：

- 启动时预创建一批连接
- 请求来了先借一个
- 用完后归还

## 连接池里有什么

- 空闲连接链表
- 互斥锁
- 信号量
- 当前空闲连接数
- 当前使用连接数
- 最大连接数

## 关键函数

### `GetInstance()`

- 作用：获取单例连接池对象

### `init(...)`

- 作用：初始化连接池
- 会创建多条 MySQL 连接并放入链表

### `GetConnection()`

- 作用：获取一个可用连接
- 用信号量表示“还剩多少连接可借”

### `ReleaseConnection(MYSQL *con)`

- 作用：归还连接

### `DestroyPool()`

- 作用：关闭所有连接

### `connectionRAII`

- 作用：构造时自动借连接，析构时自动归还

## 为什么信号量很适合连接池

因为信号量天然可以表示“有限资源数量”。

- 连接数还有余量时，线程可以获取
- 没有可用连接时，线程阻塞等待
- 连接归还后，唤醒其他等待线程

## 登录流程怎么讲

1. 浏览器提交 POST 登录表单
2. 状态机解析请求体
3. 提取用户名和密码
4. 项目启动时已经把数据库用户表加载进内存 map
5. 登录时先在 map 中匹配
6. 成功则返回 `welcome.html`
7. 失败则返回 `logError.html`

## 注册流程怎么讲

1. 解析用户名和密码
2. 先看内存 map 是否重名
3. 不重名则执行 SQL `INSERT`
4. 成功则跳转登录页
5. 失败则跳转注册失败页

## 这个实现的优点

- 演示了数据库连接池
- 登录查询很快，因为直接走内存 map
- 逻辑简单，适合教学

## 这个实现的局限

- 启动时全量加载用户，只适合小规模数据
- 密码是明文，安全性不足
- SQL 是字符串拼接，存在注入风险
- 没做真正的会话保持

## 如果用户量非常大，应该怎么优化

- 不做全量加载
- 给用户名建索引，按需查询
- 使用 Redis 做热点用户缓存
- 对用户表分库分表
- 密码改为加盐哈希存储
- 可以用布隆过滤器减少无效查询

## 如果要做登录状态保持，怎么做

推荐 `Cookie + Session`：

1. 登录成功后生成 `session_id`
2. 通过 Cookie 返回给浏览器
3. 服务端保存 `session_id -> 用户信息`
4. 后续请求根据 Cookie 找回用户状态

进一步可以把 Session 存 Redis。

## 面试里可以怎么说

> 这个项目通过 MySQL 连接池复用数据库连接，避免每次请求临时建连。登录注册逻辑上，项目启动时会把用户表加载到内存 map 中，登录时优先在内存里校验，注册时再写回数据库。这个实现适合教学演示，但如果上到大规模系统，我会把用户查询改成“索引查询 + Redis 缓存 + 安全密码存储 + Session 管理”的方案。
