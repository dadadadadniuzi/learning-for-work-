# 静态资源、项目亮点与面试表达

## `root/` 目录是干什么的

它是网站静态资源根目录。

里面包括：

- 首页和跳转页面
- 登录和注册页面
- 图片和视频展示页
- 登录成功和失败页

## 静态文件是怎么返回的

流程是：

1. URL 映射成 `root` 目录中的真实文件
2. `stat()` 检查文件是否存在、是否可读
3. `open()` 打开文件
4. `mmap()` 把文件映射到内存
5. `writev()` 一次性把响应头和文件内容发给客户端

## 为什么这算亮点

因为它不是简单地把文件一段一段 `read + write`，而是使用：

- `mmap`
- `writev`

这样更贴近高性能服务器的思路。

## 这个项目的亮点怎么概括

### 和只做 socket 收发的教学项目相比

它多做了这些：

- 引入 `epoll` 和非阻塞 I/O
- 引入线程池
- 支持 Reactor 和 Proactor 两种模式
- 加入数据库连接池
- 加入定时器
- 加入同步和异步日志
- 使用状态机解析 HTTP

## 这个项目的不足怎么说

你不要说“没有不足”，可以主动讲：

- HTTP 功能不完整
- 登录注册逻辑和网络逻辑耦合较紧
- 密码明文存储不安全
- SQL 拼接有注入风险
- 没有真正的 Session 体系
- 更偏教学型，不是工业级成品

## 30 秒介绍模板

> 这是一个基于 Linux 高并发网络编程模型实现的轻量级 C++ Web 服务器。主线程使用 epoll 监听连接和 I/O 事件，工作线程通过线程池处理请求；HTTP 报文解析采用状态机；数据库访问通过 MySQL 连接池；空闲连接通过定时器回收；日志支持同步和异步两种模式；并发模型支持 Reactor 和 Proactor。业务上实现了静态资源访问、注册和登录功能。

## 1 分钟介绍模板

> 这个项目的核心目标是把 Linux 网络编程里的关键模块串起来。整体上主线程负责监听 socket、通过 epoll 等待事件、接收新连接和分发任务；每个连接都用 http_conn 对象维护状态，内部通过状态机解析 HTTP 请求；工作线程通过线程池处理请求；数据库访问使用 MySQL 连接池复用连接；定时器负责清理长时间不活跃的连接；日志系统支持同步和异步两种写法。相比很多只做到 socket 收发的项目，它把并发模型、连接池、定时器和日志都完整串起来了。

## 面试时最值得主动强调的点

- 为什么用 `epoll`
- 为什么用状态机
- 为什么用线程池
- Reactor 和 Proactor 的区别
- 为什么数据库要连接池
- 为什么需要定时器
- 为什么异步日志更适合高并发服务器
- 为什么用 `mmap + writev`

## 最后怎么收尾最稳

> 这个项目更偏网络和并发模型展示，而不是复杂业务功能。它的价值在于把 epoll、线程池、状态机、连接池、定时器和日志这些服务器核心组件放进一个完整可运行的项目里。如果后续继续优化，我会重点补安全性、会话管理、路由解耦和更完整的 HTTP 支持。
