# TinyWebServer 10分钟速背版

> [!abstract]
> 这份笔记只保留面试最高频、最值得先背的内容。目标不是覆盖所有细节，而是让你在最短时间内把项目讲顺。

## 1. 30秒项目介绍

> 这是一个基于 Linux 高并发网络模型实现的轻量级 C++ Web 服务器。主线程使用 `epoll` 监听连接和 I/O 事件，工作线程通过线程池处理请求；HTTP 请求解析采用状态机；数据库访问通过 MySQL 连接池复用连接；空闲连接通过定时器回收；日志支持同步和异步两种模式；并发模型支持 Reactor 和 Proactor 两种实现思路。

## 2. 1分钟整体架构

项目主要分成 6 个核心模块：

- `WebServer`
  服务器总控，负责监听 socket、epoll、信号、定时器调度和事件循环
- `http_conn`
  单个连接的读写、HTTP 解析、业务处理、响应生成
- `threadpool`
  工作线程池，负责并发处理请求
- `sql_connection_pool`
  MySQL 连接池，负责数据库连接复用
- `timer`
  定时器链表，负责回收空闲连接
- `log`
  日志系统，支持同步和异步写日志

## 3. 1分钟请求处理流程

> 浏览器发起 HTTP 请求后，主线程通过 `epoll_wait` 监听事件；如果是新连接，就 `accept` 并创建 `http_conn` 对象，同时为连接绑定定时器；如果是连接上的读写事件，就根据 Reactor 或 Proactor 模式交给线程池处理；`http_conn` 用状态机解析请求行、请求头和消息体，然后根据 URL 判断是静态资源访问还是登录注册逻辑；如果需要数据库，就从连接池借一个 MySQL 连接；处理完成后构造 HTTP 响应，用 `writev` 回写给客户端；连接有活动就延长定时器，长时间无活动就关闭。

## 4. 线程池一定要会讲的点

### 为什么需要线程池

- 避免每个请求临时创建线程，减少系统开销
- 控制线程数量，防止线程过多导致频繁上下文切换
- 主线程专门做事件分发，工作线程专门做任务处理

### 工作线程是不是一直等待

是。  
线程池创建后，工作线程会一直循环：

1. 等待任务队列中的信号量
2. 被唤醒后取任务
3. 处理任务
4. 再次回到等待状态

### 工作线程处理完任务后是什么状态

不会退出，而是回到线程池继续阻塞等待下一个任务。

## 5. Reactor / Proactor 一定要会讲的点

### Reactor

- 主线程负责监听事件通知
- 工作线程负责真正的 `read/write`
- 也负责后续业务逻辑

### Proactor

- 主线程先完成实际 I/O再通知
- 工作线程只负责业务逻辑处理

### 这个项目里怎么体现

- Reactor：线程池线程里调用 `read_once()` 或 `write()`
- Proactor：主线程先 `read_once()`，线程池线程只做 `process()`

## 6. epoll 一定要会讲的点

### 为什么用 epoll

- 适合高并发
- fd 数量大时效率高
- 内核维护就绪队列，不需要每次遍历所有 fd
- 支持 LT 和 ET 模式

### 其他 I/O 复用方式

- `select`
- `poll`
- `epoll`

### 区别

- `select`：有最大 fd 数限制，且每次都要拷贝和遍历
- `poll`：没有 fd 上限，但还是线性遍历
- `epoll`：事件驱动，只返回就绪 fd，效率更高

## 7. HTTP 状态机一定要会讲的点

### 为什么要用状态机

因为 TCP 是字节流，不保证一次 `recv()` 就能读到完整 HTTP 请求。  
所以必须分阶段解析：

- 请求行
- 请求头
- 请求体

### 主状态机

- `CHECK_STATE_REQUESTLINE`
- `CHECK_STATE_HEADER`
- `CHECK_STATE_CONTENT`

### 从状态机

负责按 `\r\n` 从缓冲区里切出一行。

## 8. 登录注册逻辑一定要会讲的点

### 登录流程

1. 浏览器发送 POST 表单
2. 服务器解析请求体，取出用户名密码
3. 在内存 `map` 中校验
4. 成功则返回 `welcome.html`
5. 失败则返回 `logError.html`

### 注册流程

1. 解析用户名密码
2. 先查内存 `map` 有没有重名
3. 没有则执行 SQL `INSERT`
4. 成功后返回登录页
5. 失败返回注册失败页

## 9. 数据库连接池一定要会讲的点

### 为什么要用

- 避免每次请求都重新连接 MySQL
- 减少连接创建销毁开销
- 控制数据库连接总数

### 连接池怎么工作

- 启动时预创建多个 MySQL 连接
- 请求来了先从池里取一个连接
- 用完后归还
- 使用信号量表示剩余可用连接数

### RAII 的作用

`connectionRAII` 在构造时获取连接，在析构时自动释放连接，防止忘记归还。

## 10. 定时器一定要会讲的点

### 为什么需要定时器

为了清理长时间不活跃的连接，防止无效连接一直占用 fd 和内存资源。

### 工作原理

- 每个连接建立时绑定一个定时器
- 有读写活动时，就刷新超时时间
- `alarm + SIGALRM` 周期性触发检查
- 链表头部最早超时，超时就关闭连接

## 11. 日志系统一定要会讲的点

### 同步日志

- 当前线程直接写文件
- 实现简单
- 但会阻塞业务线程

### 异步日志

- 当前线程把日志丢进阻塞队列
- 后台线程统一写文件
- 降低主流程阻塞

### 为什么异步更适合高并发

因为磁盘 I/O 慢，异步写可以把业务线程和写盘解耦。

## 12. 静态文件返回一定要会讲的点

### 怎么返回文件

1. URL 映射到 `root` 目录中的真实文件
2. `stat()` 检查文件
3. `mmap()` 映射文件到内存
4. `writev()` 一次发送响应头和文件内容

### 为什么用 `mmap + writev`

- 减少数据拷贝
- 减少系统调用次数

## 13. HTTPS / GET / POST 速背

### HTTPS 为什么安全

- 使用 TLS/SSL 做加密
- 防止明文泄露
- 用证书验证服务器身份
- 用 MAC/摘要保证报文完整性

### HTTPS 建连过程

1. 客户端发起 TLS 握手
2. 服务端返回证书、公钥等信息
3. 客户端验证证书
4. 客户端生成对称密钥材料并加密发送
5. 双方协商出会话密钥
6. 后续使用对称加密通信

### GET 和 POST 区别

- GET 常用于获取资源，参数一般放 URL
- POST 常用于提交数据，参数一般放请求体
- GET 更适合幂等查询
- POST 更适合有副作用操作
- HTTP 协议本身对长度没有绝对限制，但工程实践里 GET 通常受 URL 长度约束

## 14. 并发压力怎么扛

如果 1000 个客户端同时访问，而线程数不多，也不意味着响应不过来。  
原因是：

- `epoll` 负责高效发现就绪连接
- 线程池不需要为每个连接都绑定一个线程
- 大量连接其实大多时间都在等待 I/O，不会一直占满 CPU

## 15. 被问到项目不足时怎么说

可以主动说这几点：

- HTTP 功能不完整，主要是教学型实现
- 登录注册逻辑和网络逻辑耦合较紧
- 密码明文存储，安全性不足
- SQL 拼接有注入风险
- 用户状态没有做完整会话管理
- 定时器和路由系统都还有优化空间

## 16. 最后 10 句救命答案

1. 这是一个基于 `epoll + 线程池 + 非阻塞 I/O` 的轻量级 Web 服务器。
2. 主线程负责事件监听和分发，工作线程负责请求处理。
3. `http_conn` 表示一个客户端连接，内部维护 HTTP 解析状态。
4. 项目用状态机解析 HTTP，因为 TCP 是流式传输。
5. 项目支持 Reactor 和 Proactor 两种并发模型。
6. MySQL 连接池通过复用连接减少数据库访问开销。
7. 定时器负责关闭长时间不活跃的连接。
8. 日志系统支持同步和异步两种模式，异步通过阻塞队列实现。
9. 静态文件使用 `mmap + writev` 返回，提高传输效率。
10. 项目更偏重网络和并发模型展示，而不是复杂业务功能。
