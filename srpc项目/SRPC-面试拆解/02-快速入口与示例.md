## `echo_pb.proto` 说明了什么

它定义了四件事：

1. 请求结构 `EchoRequest`
2. 响应结构 `EchoResponse`
3. 服务 `Example`
4. 方法 `Echo`

也就是：

“客户端给 `Example` 服务发一个 `Echo` 请求，类型是 `EchoRequest`，服务端回一个 `EchoResponse`。”

## 代码生成之后会得到什么

执行：

```bash
protoc echo_pb.proto --cpp_out=./ --proto_path=./
srpc_generator protobuf ./echo_pb.proto ./
```

会生成一个 `echo_pb.srpc.h`，里面通常包含：

1. `Example::Service`
2. `Example::SRPCClient`
3. 其他协议客户端类
4. 同步、异步、future、task 风格接口

## 服务端 demo 在做什么

文件：[tutorial-01-srpc_pb_server.cc](G:\计算机学习\srpc项目\srpc-master\tutorial\tutorial-01-srpc_pb_server.cc)

核心步骤：

1. 继承生成的 `Example::Service`
2. 重写 `Echo(EchoRequest *, EchoResponse *, RPCContext *)`
3. 创建 `SRPCServer`
4. `add_service(&impl)`
5. `start(1412)`

业务函数本身非常简单：

1. 读取 request
2. 填 response
3. 可选地从 `ctx` 取状态或附加信息

## 客户端 demo 在做什么

文件：[tutorial-02-srpc_pb_client.cc](G:\计算机学习\srpc项目\srpc-master\tutorial\tutorial-02-srpc_pb_client.cc)

核心步骤：

1. 创建 `Example::SRPCClient`
2. 构造请求对象
3. 发异步调用 `client.Echo(&req, callback)`
4. 也支持同步调用 `client.Echo(&req, &resp, &sync_ctx)`

## 从 demo 应该看出什么

面试时要强调：

1. 业务开发者只关心 `request / response / context`
2. 底层 socket、包头、重试、压缩都已经被框架封装
3. 这正是代码生成器和运行时框架配合的价值
