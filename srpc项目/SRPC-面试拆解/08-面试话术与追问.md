# SRPC 面试话术与追问

返回：[[SRPC-面试拆解笔记]]

## 30 秒版本

> 这个项目是一个基于 Workflow 的 C++ RPC 框架。它支持 protobuf 和 thrift 作为 IDL，通过代码生成器把接口定义生成成客户端和服务端代码。底层统一处理序列化、压缩、协议封装、网络传输和异步任务模型，同时支持 SRPC、BRPC、Thrift、TRPC，以及 HTTP+JSON 访问。服务端通过 service 和 method 做分发，客户端通过 RPCClientTask 发起调用，trace 和 metrics 则通过模块和 filter 机制非侵入接入。*

## 2 分钟版本

> 我理解这个项目的核心不是单个业务服务，而是一套 RPC 基础设施。它先通过 `.proto` 或 `.thrift` 定义服务，再由 `srpc_generator` 生成 `.srpc.h`，里面包含服务端基类和客户端类。业务方只要继承生成的 `Service` 类，重写像 `Echo(request, response, ctx)` 这样的函数，就能实现服务端。客户端则直接调用生成的 `client.Echo(...)`。  
> 内部实现上，它先把请求对象序列化成 protobuf、thrift 或 json，再根据需要做 gzip、snappy 等压缩，然后封装成 SRPC 或 HTTP 消息，通过 Workflow 的网络 task 发送出去。服务端收到包后，先反序列化 meta，再找到对应 service 和 method，解压请求体，执行过滤器，再调用真实业务函数。返回时再反向做序列化、压缩和协议封装。  
> 这个项目设计比较好的地方是分层解耦、模板复用，以及通过模块系统把 trace、metrics 等横切能力独立出来。*

## 高频追问

### 为什么要代码生成

回答思路：

1. 原生 protobuf/thrift RPC 接口对业务方不够友好
2. 生命周期和上下文不好统一
3. 框架想统一暴露简单的 `request/response/context` 模型

### 为什么要支持 HTTP+JSON

回答思路：

1. 便于跨语言访问
2. 便于调试
3. 能用 curl、脚本、浏览器快速验证

### 为什么需要 `RPCContext`

回答思路：

1. 统一拿状态码、错误、远端 IP、header
2. 统一写附加信息、baggage、日志
3. 能拿到 `SeriesWork`，继续挂 Workflow 任务

### 为什么要有 `rpc_types.h`

回答思路：

1. 这是协议 traits 层
2. 用来声明不同协议的 req/resp 和默认行为
3. 让模板化 client/server 复用同一套主流程

## 最容易说错的地方

1. 不要把 `Workflow` 说成业务框架
2. 不要把 `SRPC` 和 `protobuf` 混为一谈
3. 不要说这个项目只是 socket 封装
4. 不要说所有协议都是各写一套
5. 不要说服务端收到包就直接回业务函数，没有中间流程

## 最后冲刺

如果今晚只背 5 个点，就背：

1. 这是 RPC 基础设施，不是业务 demo
2. 核心链路是 `RPCClientTask -> RPCServer::server_process -> RPCServerTask -> RPCClientTask::rpc_callback`
3. 代码生成器是用户体验核心
4. `rpc_types.h` 是多协议复用关键
5. `RPCModule + RPCFilter` 是 trace/metrics 扩展关键
