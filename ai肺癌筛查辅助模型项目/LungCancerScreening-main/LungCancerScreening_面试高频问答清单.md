# LungCancerScreening 面试高频问答清单

> [!summary]
> 这份笔记的目标不是“完整讲项目”，而是“帮你面试时快速答题”。
> 你可以把它当成面试前速背提纲。

## 1. 一句话介绍项目

### Q：你先简单介绍一下这个项目。

A：

这是一个 AI 辅助肺癌筛查系统，输入是 CT 扫描文件，输出是结节检测、真假结节判断和恶性风险评估结果。整体采用三阶段流程：先用 2D UNet 做切片分割找候选区域，再用 3D CNN 判断候选区域是否是真结节，最后再用另一个 3D CNN 判断结节的良恶性风险，并通过 Flask 和 React 提供可视化服务。

## 2. 项目目标

### Q：这个项目到底解决什么问题？

A：

它想解决的是“如何从肺部 CT 中自动找出可疑结节，并给出风险判断”。它不是直接输出一个很抽象的癌症结论，而是尽量模拟临床辅助流程：先找可疑位置，再判断是不是结节，再评估风险高低。这样更可解释，也更容易落地成一个辅助系统。

### Q：为什么你们不直接做一个端到端模型？

A：

因为端到端虽然听起来更简洁，但在这个任务里不一定更稳定。这个项目拆成三阶段以后，每一阶段职责都很清楚：

1. 分割负责找位置。
2. 分类负责过滤误报。
3. 恶性模型负责风险分析。

这样更容易定位问题，也更符合真实业务流程。

## 3. 数据部分

### Q：你们用的是什么数据集？

A：

主要用的是 LUNA16 数据集。项目里依赖 `.mhd/.raw` 格式的 CT 数据，以及 `annotations.csv`、`candidates.csv`、`annotations_with_malignancy.csv` 这些标注文件。

### Q：数据在项目里是怎么组织的？

A：

原始 CT 数据放在 `data-unversioned/data/subset*/` 下，标注 CSV 在 `data/` 目录。训练时会先读标注，再把病人坐标转为体素坐标，然后从完整 CT 中裁出固定大小的 2D 或 3D 样本。

### Q：为什么要做坐标转换？

A：

因为标注文件里的结节中心通常是病人坐标系下的 `(x, y, z)`，而模型训练和裁剪图像时用的是数组索引，也就是 `(index, row, col)`。所以项目里专门写了 `patientCoord2voxelCoord` 和 `voxelCoord2patientCoord` 来做坐标系转换。

## 4. 分割模型

### Q：第一阶段为什么用 UNet？

A：

因为 UNet 非常适合医学图像分割。它的编码器负责提取特征，解码器负责恢复空间分辨率，跳跃连接可以保留细节信息，比较适合做“找到 CT 上哪里有疑似结节”这个任务。

### Q：分割模型输入是什么？

A：

输入不是单张切片，而是 7 通道切片，也就是当前切片前后各 3 张，再加当前切片本身。这样做是为了引入上下文，因为肺结节是 3D 结构，只看一张切片很容易误判。

### Q：分割模型输出是什么？

A：

输出是一张二值分割概率图，表示当前切片每个像素属于可疑区域的概率。之后会用阈值把它转成 mask。

### Q：分割训练用的损失函数是什么？

A：

主要是 Dice Loss，而且代码里还额外对假阴性损失做了加权。意思是作者更重视“不要漏掉结节”，哪怕多报一点也比漏报更可接受。

### Q：为什么分割结果之后还要做连通域分析？

A：

因为分割阶段输出的是像素级 mask，但分类模型需要的是一个个候选结节中心。所以项目会把分割得到的 3D mask 做连通域分析，把每个连通块转成一个候选区域。

## 5. 真假结节分类

### Q：第二阶段为什么还要做一个分类模型？

A：

因为分割模型为了提高召回，通常会产生很多误报，比如血管、噪声、其他组织都可能被当成可疑区域。所以需要一个 3D CNN 来对候选区域做二次筛选，判断它到底是不是真结节。

### Q：这个 3D CNN 输入是什么？

A：

输入是从完整 CT 中以候选点为中心裁出的一个固定大小 3D 体块，尺寸大致是 `32 x 48 x 48`，再加一个通道维，所以张量形状通常是 `[B, 1, 32, 48, 48]`。

### Q：它的输出是什么？

A：

输出两类 logits 和 softmax 概率：

1. 不是结节
2. 是结节

### Q：为什么这里要用 3D CNN？

A：

因为真假结节判断不仅依赖某一张切片的纹理，还依赖整个空间形状。3D CNN 能更好地学习结节的立体结构特征。

## 6. 良恶性分类

### Q：第三阶段和第二阶段的区别是什么？

A：

结构上非常接近，都是输入一个 3D 候选块，但标签不一样。第二阶段判断“是不是结节”，第三阶段判断“这个结节更偏良性还是恶性”。

### Q：良恶性模型是从零训练的吗？

A：

不是完全从零。这个项目用了微调思路，从已有的 3D 结节分类模型出发，只解冻后面几层做 fine-tune。这样可以复用前面已经学到的 3D 结构特征。

### Q：为什么微调而不是完全重新训练？

A：

因为良恶性任务的数据通常比真假结节任务更少，直接从零训练容易不稳定。微调可以让训练更快，也更容易收敛。

## 7. 数据增强与缓存

### Q：项目里做了哪些数据增强？

A：

主要有：

1. 翻转
2. 偏移
3. 缩放
4. 旋转
5. 加噪声

分割任务是 2D 仿射增强，分类任务是 3D 仿射增强。

### Q：为什么这个项目要做缓存？

A：

因为医学 CT 很大，每次训练时如果都从完整体数据里重新裁块，会很慢。项目里用 `diskcache + gzip` 把常用样本缓存到磁盘，大大减少重复预处理开销。

### Q：缓存脚本的作用是什么？

A：

像 `prepcache.py`、`prep_seg_cache.py`、`TumorPrepCache.py` 这些脚本会先把整个数据集过一遍，提前把样本写进缓存，后面正式训练时读取速度更快。

## 8. 训练流程

### Q：训练是怎么组织的？

A：

每个任务基本都分两步：

1. 先预热缓存。
2. 再启动训练。

训练时会创建训练集和验证集，跑多个 epoch，记录 TensorBoard 日志，按指标保存 checkpoint 和 best model。

### Q：真假结节分类看什么指标？

A：

主要看：

1. loss
2. accuracy
3. precision
4. recall
5. F1

因为正负样本不平衡，所以 F1 比单纯 accuracy 更有意义。

### Q：良恶性分类看什么指标？

A：

除了 precision、recall、F1 以外，还会看 AUC，并绘制 ROC 曲线。因为良恶性判断更像风险评分任务，所以 AUC 很重要。

## 9. 整体推理链路

### Q：完整推理流程怎么讲？

A：

完整流程是：

1. 用户上传 `.mhd` 和 `.raw`
2. 后端读取 CT 数据
3. 分割模型逐切片做候选区域分割
4. 对分割 mask 做连通域分析
5. 生成候选结节列表
6. 对每个候选结节裁 3D 小块
7. 真假结节分类模型判断是不是结节
8. 恶性模型给出风险概率
9. 后端汇总结果，生成总体结论
10. 保存数据库并返回前端展示

### Q：总体结论是怎么给出的？

A：

系统会看检测出的结节里最可疑的那个，如果恶性风险高，就把整体结果标成 `high_risk`；如果没有结节，就是 `no_nodules_found`；如果有结节但风险低或中等，就分别给出 `low_risk` 或 `moderate_risk`。

## 10. 评估脚本

### Q：`model_evel.py` 是干什么的？

A：

它不是训练脚本，而是完整 pipeline 的联合评估脚本。它会把三个模型串起来，在验证集上跑整条流程，然后把预测结果和真实标注做距离匹配，最后输出混淆矩阵。

### Q：为什么要单独写一个联合评估脚本？

A：

因为单个模型的好成绩不代表系统整体效果好。比如分割 recall 很高，但如果误报太多，后面的分类器压力会很大。所以必须评估完整链路，而不是只看单点模型。

## 11. 后端设计

### Q：后端是怎么组织的？

A：

后端基于 Flask，核心类是 `TumorPredictionSystem`，它封装了模型加载和完整推理流程。外层再通过多个 API 提供健康检查、上传、历史记录、统计、切片图像、聊天问答等服务。

### Q：数据库里存了什么？

A：

数据库里有一个 `Diagnosis` 表，主要存：

1. 原始文件名
2. 上传后的文件路径
3. 时间戳
4. 最终诊断结果
5. 置信度
6. 完整 JSON 结果

### Q：`/api/upload` 做了什么？

A：

它会校验上传的 `.mhd/.raw` 文件是否匹配，然后调用完整推理流程，最后把结果写数据库，并把诊断 JSON 返回给前端。

### Q：`/api/ct-slice` 有什么作用？

A：

它可以动态返回某一张 CT 切片的 PNG 图像，并根据前端传来的结节信息，在图片上画边界框。这个接口支撑了前端 CT 浏览器。

## 12. 前端设计

### Q：前端有哪些页面？

A：

主要有五个页面：

1. Dashboard 仪表盘
2. Upload 上传页
3. History 历史记录页
4. ModelStatus 模型状态页
5. Chat AI 问答页

### Q：上传页的核心流程是什么？

A：

用户选 `.mhd` 和 `.raw` 文件后，前端先校验文件名前缀是否一致，然后发给 `/api/upload`。后端返回结果后，前端会展示总结论、最可疑结节、结节列表，并支持打开 CTViewer 查看切片图像。

### Q：History 页面除了看历史还有什么亮点？

A：

它不只是看表格，还能点开详细报告，并把某份报告带着 `predictionId` 跳转到聊天页，这样用户可以针对同一份报告继续问 AI。

### Q：Chat 页面是怎么做上下文问答的？

A：

前端会把 `predictionId` 一起传给后端，后端再去数据库取出该报告的摘要内容，把报告上下文和用户问题一起拼到 prompt 里发给大模型。

## 13. 工程亮点

### Q：你觉得这个项目最有亮点的地方是什么？

A：

我觉得有四点：

1. 它不是单个模型，而是完整系统，从训练、评估到前后端全打通。
2. 三阶段设计比较清楚，逻辑上符合临床辅助流程。
3. 有缓存、TensorBoard、模型持久化这些工程化细节。
4. 有可视化 CT 浏览器和报告问答，产品感比较强。

## 14. 不足与优化

### Q：这个项目有哪些不足？

A：

有几个比较明显的点：

1. 存在历史遗留代码，比如个别函数和类定义不完全一致。
2. 有硬编码路径、硬编码数据库配置。
3. 一部分统计数据是模拟值，不是真实监控。
4. 文件和页面里有少量中文编码异常。
5. 缺少更完整的自动化测试和统一配置管理。

### Q：如果继续优化，你会怎么做？

A：

我会从这几个方向优化：

1. 做统一配置文件，移除硬编码。
2. 清理遗留逻辑，统一命名和目录结构。
3. 增加自动化测试和推理链路测试。
4. 增加真实监控和日志系统。
5. 如果追求精度，可以尝试更强的 3D backbone 或多阶段后处理优化。

## 15. 你在面试里可以主动补充的细节

### Q：你在读这个项目时注意到了什么细节？

A：

我注意到这个项目很强调“召回率优先”的思路，比如分割任务里对假阴性损失做了额外加权，这说明作者更担心漏检结节。另外整个项目里对坐标转换、候选块裁剪、缓存和多阶段评估都做了比较明确的封装，这说明它不是只想做一个实验模型，而是想往工程系统方向推进。

## 16. 最后一分钟速背

### Q：如果时间只够你讲一分钟，你会怎么讲？

A：

这是一个基于 LUNA16 的 AI 辅助肺癌筛查系统。整体分三步：先用 2D UNet 对 CT 切片做分割找出疑似结节区域，再把候选区域裁成 3D 小块，用 3D CNN 判断是不是真结节，最后再做良恶性风险评估。训练部分包括数据集构建、坐标转换、缓存、数据增强、TensorBoard 记录和模型保存；推理部分通过 Flask 暴露 API，并接了一个 React 前端，支持上传 CT、查看切片、保存历史和 AI 问答。这个项目的优点是全链路完整、可解释性强，但也存在一些硬编码和遗留代码，后续可以继续做工程化优化。
