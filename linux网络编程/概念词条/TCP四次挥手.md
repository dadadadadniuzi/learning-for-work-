# TCP四次挥手

TCP 四次挥手是 [[linux网络编程/概念词条/TCP|TCP]] 关闭连接时双方释放连接的过程。它发生在应用层调用 `close`、`shutdown` 或对端关闭连接之后。

## 为什么通常是四次

TCP 是全双工通信，双方都有独立的发送方向和接收方向。一方说“我不发了”，并不代表另一方也立刻没有数据要发，所以关闭连接通常需要分别关闭两个方向。

这也是 [[linux网络编程/概念词条/TCP半关闭|TCP半关闭]] 能存在的原因：一方可以先关闭写方向，但仍然继续读对方的数据。

## 四次过程

| 次数 | 方向 | 报文标志 | 含义 |
|---|---|---|---|
| 第一次 | 主动关闭方 -> 被动关闭方 | `FIN` | 主动关闭方表示自己已经没有数据要发送。 |
| 第二次 | 被动关闭方 -> 主动关闭方 | `ACK` | 被动关闭方确认收到关闭请求，但自己可能还有数据要发。 |
| 第三次 | 被动关闭方 -> 主动关闭方 | `FIN` | 被动关闭方也发送完数据，请求关闭自己的发送方向。 |
| 第四次 | 主动关闭方 -> 被动关闭方 | `ACK` | 主动关闭方确认对方的关闭请求，连接最终释放。 |

## 和 socket 函数的关系

- 一端调用 `close` 后，通常会触发 FIN。
- 一端调用 [[linux网络编程/函数笔记/Socket/shutdown|shutdown]] 并选择 `SHUT_WR` 时，也会关闭写方向并触发 FIN，这就是常见的 [[linux网络编程/概念词条/TCP半关闭|TCP半关闭]]。
- 另一端继续调用 [[linux网络编程/函数笔记/Socket/recv|recv]] 或 `read`，读到返回值 `0`，表示对端已经正常关闭发送方向。
- 如果程序还想区分“只关闭写方向”和“完全关闭”，可以学习 `shutdown`，它比 `close` 更细。

## 状态变化速记

- 主动关闭方常见状态：`ESTABLISHED -> FIN_WAIT_1 -> FIN_WAIT_2 -> TIME_WAIT -> CLOSED`。
- 被动关闭方常见状态：`ESTABLISHED -> CLOSE_WAIT -> LAST_ACK -> CLOSED`。
- [[linux网络编程/概念词条/TIME_WAIT|TIME_WAIT]] 用来确保最后一个 ACK 有机会重传，也让旧连接中的延迟报文自然过期；等待时间通常按 [[linux网络编程/概念词条/MSL|2MSL]] 理解。

## 易错点

- 四次挥手不一定总是严格表现为四个独立包，某些 ACK 和 FIN 可能被合并，但逻辑上仍然是两个方向分别关闭。
- `recv/read` 返回 `0` 不是错误，而是对端正常关闭连接。
- 主动关闭方进入 [[linux网络编程/概念词条/TIME_WAIT|TIME_WAIT]] 是正常现象，不是程序卡死。
