# SRPC 10分钟速背版

返回：[[SRPC-面试拆解笔记]]  
关联：[[SRPC-面试拆解/01-项目总览]] | [[SRPC-面试拆解/04-请求完整链路]] | [[SRPC-面试拆解/08-面试话术与追问]]

> [!abstract]
> 这篇只保留明天面试前最值得背的内容。目标不是看懂所有源码，而是让你能在短时间内稳定讲清楚项目。

## 1. 一句话介绍

**这是一个基于 Workflow 的 C++ RPC 框架，用 protobuf/thrift 定义服务，通过代码生成器生成 client/server 代码，底层统一处理序列化、压缩、协议封装、网络传输、异步任务、trace 和 metrics。**

## 2. 这个项目解决什么问题

1. 业务开发不用自己写 socket、收发包、协议解析。
2. IDL 定义可以直接生成可用的客户端和服务端接口。
3. 同一套业务逻辑能跑在多种协议和传输方式上。
4. trace、metrics、filter 可以非侵入接入。

## 3. 最核心的 5 个关键词

1. `代码生成`
2. `分层解耦`
3. `Workflow 异步任务`
4. `多协议复用`
5. `模块化扩展`

## 4. 你必须会讲的整体链路

### 客户端

1. 调 `client.Echo(&req, callback)`
2. 生成 `RPCClientTask`
3. 序列化 request
4. 压缩 body
5. 序列化 meta
6. Workflow 发出去

### 服务端

1. `RPCServer::new_session()` 创建 `RPCServerTask`
2. `server_process()` 解析 meta
3. 根据 `service_name` 找 service
4. 根据 `method_name` 找 method
5. 解压 body
6. 执行 filter/module begin
7. 调业务函数

### 回包

1. `RPCServerTask::message_out()`
2. 序列化 response
3. 压缩
4. 序列化 meta
5. 执行 module end
6. 回给客户端

### 客户端回调

1. `rpc_callback()` 解压响应
2. 反序列化 response
3. 调用户 callback
4. `RPCContext` 提供状态和错误信息

## 6. 你必须记住的 6 个核心类

1. `RPCService`
   负责服务和方法注册。
2. `RPCServer`
   负责接收请求、查找 service/method、驱动服务端处理流程。
3. `RPCClient`
   负责创建 RPC 调用任务。
4. `RPCContext`
   负责保存这次调用的上下文。
5. `RPCClientTask`
   负责客户端请求生命周期。
6. `RPCServerTask`
   负责服务端响应生命周期。

## 7. 为什么这个项目设计得好

1. **分层清晰**
   IDL、序列化、压缩、协议、传输、业务逻辑分开。
2. **复用高**
   通过模板和 `rpc_types.h` 支持多协议复用。
3. **异步能力强**
   基于 Workflow，RPC 本身就是 task。
4. **扩展友好**
   trace、metrics、custom filter 都是模块化接入。

## 8. 为什么要代码生成

直接答：

1. 原生 protobuf/thrift RPC 接口对业务方不够友好。
2. 框架想统一暴露更简单的 `request/response/context` 编程模型。
3. 代码生成能自动生成 client/server skeleton，降低接入成本。

## 9. 为什么支持 HTTP+JSON

直接答：

1. 便于跨语言访问。
2. 便于调试。
3. 可以直接用 curl 或脚本验证服务。

## 10. 为什么需要 `RPCContext`

直接答：

1. 统一拿状态码、错误码、远端 IP、header。
2. 服务端可以设置响应格式、压缩方式、header。
3. 可以拿到 `SeriesWork`，继续往 Workflow 任务流里挂任务。

## 11. 为什么 `rpc_types.h` 很重要

直接答：

它是协议 traits 层。  
不同协议只需要声明请求类型、响应类型、默认数据格式和少量差异逻辑，就能复用整个 `RPCClient`/`RPCServer` 主框架。

## 12. 最容易被问的“从零实现”

标准答案：

> 我会先定义统一抽象：request、response、message、context、service、client、server。然后实现消息编解码、压缩和协议封装，再把请求挂到 Workflow 的异步 task 生命周期上。之后做服务注册和方法分发，再做代码生成器，把 proto/thrift 生成成业务可直接使用的接口。最后补 trace、metrics、custom filter 等横切模块，并通过 traits 支持多协议复用。*

## 13. 30秒版本

> 这是一个基于 Workflow 的 C++ RPC 框架。它支持 protobuf 和 thrift 作为 IDL，通过代码生成器生成客户端和服务端接口，底层统一处理序列化、压缩、协议封装、网络传输和异步任务模型，同时支持多种 RPC 协议以及 HTTP+JSON 访问。服务端通过 service 和 method 分发调用，trace 和 metrics 通过模块化方式接入。*

## 14. 2分钟版本

> 这个项目本质上是一套 RPC 基础设施，不是单个业务 demo。它先通过 `.proto` 或 `.thrift` 定义服务，再由 `srpc_generator` 生成 `.srpc.h`，里面包含服务端基类和客户端类。业务方只要继承生成的 `Service`，实现像 `Echo(request, response, ctx)` 这样的函数，就能完成服务端逻辑。客户端则直接调用生成的 `client.Echo(...)`。  
> 内部实现上，请求会先被序列化成 protobuf、thrift 或 json，再根据配置做 gzip、snappy 等压缩，然后封装成 SRPC 或 HTTP 消息，通过 Workflow 网络任务发送。服务端收到后，先解析 meta，再找到 service 和 method，解压请求、执行 filter、调用业务函数，最后把 response 反向序列化并回传。这个项目设计好的地方在于分层解耦、模板复用，以及模块化扩展。*

## 15. 最后 1 分钟只背这 8 句

1. 这是 RPC 基础设施，不是业务 demo。
2. 底层基于 Workflow，所以天然异步。
3. 用 protobuf/thrift 定义服务。
4. 用代码生成器生成 `.srpc.h`。
5. 客户端核心是 `RPCClientTask`。
6. 服务端核心是 `RPCServer::server_process()`。
7. 多协议复用靠 `rpc_types.h`。
8. trace 和 metrics 扩展靠 `RPCModule + RPCFilter`。
