# 线程池

## 它是什么

线程池是提前创建一组工作线程，让它们长期等待任务；当有新任务到来时，把任务放入任务队列，由空闲线程取出并执行。

在网络服务器中，线程池常用于优化“每来一个客户端就创建一个线程”的模型：主线程负责 [[linux网络编程/函数笔记/Socket/accept|accept]] 新连接，把已连接 socket 或业务任务投递到队列，工作线程从队列中取任务并处理通信。

## 为什么需要线程池

最直接的多线程服务器模型是“一个连接一个线程”：

```text
accept 一个客户端
  -> pthread_create 创建一个线程
  -> 线程处理该客户端
  -> 线程退出
```

这个模型好理解，但连接很多时会有问题：

- 频繁创建和销毁线程有开销。
- 线程数量不受控制，客户端很多时可能耗尽系统资源。
- 每个线程都占用栈空间和调度资源。
- 高并发下线程切换会变多，性能反而下降。

线程池的思想是：线程先创建好，任务来了复用已有线程，避免无限制创建线程。

## 核心组成

线程池通常由这些部分组成：

- 工作线程数组：提前创建的一组线程，通常用 [[linux系统编程/函数笔记/线程/pthread_create|pthread_create]] 创建。
- 任务队列：保存待处理任务，例如已连接 fd、函数指针、业务参数等。
- 互斥锁：保护任务队列，常用 [[linux系统编程/概念词条/pthread_mutex_t|pthread_mutex_t]]。
- 条件变量：队列为空时让工作线程睡眠，任务到来时唤醒，常用 [[linux系统编程/概念词条/pthread_cond_t|pthread_cond_t]]。
- 关闭标志：通知线程池退出，避免程序结束时线程无法收尾。

## 工作流程

```text
主线程
  socket / bind / listen
  循环 accept
  把 cfd 封装成任务
  加入任务队列
  唤醒一个工作线程

工作线程
  循环等待任务
  从任务队列取出 cfd
  read / recv 读取请求
  send / write 发送响应
  close cfd
  继续等待下一个任务
```

## 生产者消费者模型

线程池本质上很像生产者消费者模型：

- 生产者：主线程或事件循环，把新连接或业务任务放入任务队列。
- 消费者：工作线程，从任务队列中取任务执行。
- 共享缓冲区：任务队列。
- 同步手段：互斥锁保护队列，条件变量负责等待和唤醒。

典型伪代码：

```c
pthread_mutex_lock(&pool->mutex);

while (queue_empty(pool->queue) && !pool->shutdown) {
    pthread_cond_wait(&pool->cond, &pool->mutex);
}

task = queue_pop(pool->queue);

pthread_mutex_unlock(&pool->mutex);

handle_task(task);
```

## 为什么条件变量要配合互斥锁

任务队列是多个线程共享的数据结构。

- 主线程会向队列里添加任务。
- 工作线程会从队列里取任务。
- 多个工作线程可能同时竞争同一个任务。

所以访问队列时必须加锁。条件变量只负责“没有任务时睡眠，有任务时唤醒”，它不负责保护队列本身，因此要和 [[linux系统编程/概念词条/pthread_mutex_t|pthread_mutex_t]] 配合。

等待条件变量时通常写 `while`，不是 `if`：

```c
while (queue_empty(pool->queue)) {
    pthread_cond_wait(&pool->cond, &pool->mutex);
}
```

原因是线程被唤醒后不代表一定有任务，可能发生虚假唤醒，也可能任务已经被其他线程抢走。

## 线程池和多线程并发服务器的关系

[[linux网络编程/概念词条/多线程并发服务器|多线程并发服务器]] 可以有两种常见写法：

- 简单版：每来一个连接就创建一个线程。
- 线程池版：提前创建固定数量线程，新连接作为任务交给线程池处理。

线程池版更适合长时间运行的服务器，因为它控制了线程数量，并减少了线程创建销毁的开销。

## 线程池和 IO 多路复用的关系

线程池和 [[linux网络编程/概念词条/IO多路复用|IO多路复用]] 不是互斥关系。

常见组合方式：

- 单 reactor + 线程池：主线程用 [[linux网络编程/概念词条/epoll模型|epoll]] 监听连接和读写事件，把耗时业务交给线程池。
- 主线程 accept + 线程池：主线程只接收新连接，工作线程负责阻塞式通信。
- 多线程各自 epoll：更复杂，每个线程维护自己的事件循环。

学习阶段先记住：IO 多路复用偏向“一个线程管理很多 fd”，线程池偏向“多个固定工作线程处理任务”。

## 优点

- 减少频繁创建和销毁线程的开销。
- 限制并发线程数量，避免线程无限增长。
- 任务队列可以削峰，短时间大量任务不会立刻创建大量线程。
- 结构更适合长期运行的服务器程序。

## 缺点

- 实现复杂度比“每连接一个线程”高。
- 需要正确处理锁、条件变量、任务队列、退出流程。
- 如果任务执行时间太长，可能导致队列堆积。
- 线程数设置不合理会影响性能：太少处理不过来，太多调度开销大。

## 常见设计字段

一个简化线程池结构可能包含：

```c
typedef struct ThreadPool {
    pthread_t *threads;
    int thread_count;

    TaskQueue queue;
    pthread_mutex_t mutex;
    pthread_cond_t cond;

    int shutdown;
} ThreadPool;
```

任务结构可能是：

```c
typedef struct Task {
    int fd;
    void (*handler)(int fd);
} Task;
```

真实项目中任务结构可能更复杂，例如包含客户端地址、请求缓冲区、业务上下文等。

## 易错点

- 访问任务队列必须加锁，否则多个线程同时修改队列会产生竞态。
- 条件变量等待要放在 `while` 中，不要只用 `if`。
- 线程池销毁时要设置关闭标志并唤醒所有线程，否则工作线程可能一直睡眠。
- 工作线程处理完连接后要关闭对应 fd，避免文件描述符泄漏。
- 线程池不是线程越多越好，线程数通常要结合 CPU 核数、任务类型和阻塞情况调整。
