# Webbench压力测试与性能瓶颈分析

返回：[[TinyWebServer-面试拆解笔记]]

相关：[[TinyWebServer-拆解/05-线程池与并发模型]]、[[TinyWebServer-拆解/09-静态资源、项目亮点与面试表达]]、[[TinyWebServer-面试问答模拟版]]

> [!abstract]
> 这篇专门讲 4 件事：
> 1. Webbench 是什么
> 2. 如何对 TinyWebServer 做压力测试
> 3. 压测时看哪些指标
> 4. 如何根据测试结果分析服务端性能瓶颈

## 1. 什么是 Webbench

Webbench 是一个轻量级 Web 压力测试工具。  
它的目标很直接：

- 模拟多个客户端同时访问服务器
- 在一定时间内持续发送 HTTP 请求
- 统计服务器能处理多少请求、传输了多少数据、失败了多少次

一句话理解：

> Webbench 就是“用程序批量模拟很多个浏览器去打你的服务器，然后统计服务器扛住了多少请求”。

## 2. Webbench 在这个项目里为什么合适

这个项目本身就是一个轻量级 HTTP 服务器，核心关注的是：

- 并发连接处理能力
- I/O 模型差异
- 线程池调度能力
- 日志、数据库连接池、定时器等附加模块对性能的影响

而 Webbench 正好擅长测：

- 高并发 HTTP 请求吞吐
- 不同配置下的请求处理效率
- 同一服务在不同参数下的相对差异

也就是说，它非常适合做这个项目的“并发性能对比实验”。

## 3. Webbench 的工作原理是什么

从 `webbench.c` 的实现可以看出，它的大致流程是：

1. 解析命令行参数
2. 构造 HTTP 请求字符串
3. 启动多个并发客户端进程
4. 每个客户端不断向目标服务器发起请求
5. 在设定时间内持续循环
6. 统计：
   - 成功请求数
   - 失败请求数
   - 传输字节数
7. 最后汇总结果并输出

## 4. 它本质上在测什么

本质上测的是：

- 服务器单位时间内能完成多少 HTTP 请求
- 服务器在一定并发下是否稳定
- 在请求持续打满时，哪些模块开始成为瓶颈

所以它不只是“跑个数字”，而是在帮你回答：

- 这个服务器快不快
- 稳不稳
- 卡在哪里

## 5. 常用命令参数是什么

项目自带说明里最常见的形式是：

```bash
webbench -c 500 -t 30 http://127.0.0.1/index.html
```

常用参数：

- `-c`
  并发客户端数
- `-t`
  压测持续时间，单位秒

从 `webbench.c` 里还能看到一些扩展参数：

- `-f` / `--force`
  不等待服务器响应，适合测极限发送能力
- `-r` / `--reload`
  强制重新加载
- `-9`
  使用 HTTP/0.9
- `-1`
  使用 HTTP/1.0
- `-2`
  使用 HTTP/1.1
- `--get`
  GET 请求
- `--head`
  HEAD 请求
- `--options`
  OPTIONS 请求
- `--trace`
  TRACE 请求

不过在这个项目里，面试最常讲的是：

- `-c` 并发量
- `-t` 持续时间

## 6. 对 TinyWebServer 怎么做压力测试

## 6.1 第一步：先启动服务器

例如：

```bash
./server
```

或者按指定参数启动：

```bash
./server -p 9006 -l 0 -m 3 -o 1 -s 8 -t 8 -c 1 -a 0
```

这里建议压测前先明确好你要测的维度，比如：

- Reactor 还是 Proactor
- LT 还是 ET
- 是否开启日志
- 线程数多少
- 数据库连接池大小多少

## 6.2 第二步：尽量减少干扰因素

为了得到更纯粹的服务器吞吐数据，通常会：

- 关闭日志
- 选一个稳定的静态资源地址
- 保证服务器和压测环境尽量稳定

原因是：

- 日志会引入额外磁盘 I/O
- 动态业务可能引入数据库波动
- 文件大小不同也会显著影响吞吐

## 6.3 第三步：运行 Webbench

例如：

```bash
webbench -c 500 -t 30 http://127.0.0.1:9006/judge.html
```

表示：

- 同时模拟 500 个客户端
- 连续压 30 秒
- 持续请求 `judge.html`

## 6.4 第四步：记录结果

你需要重点记录：

- 并发数
- 压测时长
- 请求资源类型
- 服务器启动参数
- 测试输出结果

如果只是记一个“QPS 很高”，面试时其实不好用。  
真正有说服力的是：

> 在某种参数组合下，用多少并发、压多久、请求什么资源、最终得到什么结果。

## 7. 压测时最重要的指标是什么

## 7.1 吞吐量 / 请求率

最常见的就是：

- QPS
  每秒请求数
- pages/min
  每分钟处理页面数

它表示：

- 单位时间内服务器完成了多少个请求

这个指标最能反映整体吞吐能力。

## 7.2 数据传输速率

例如：

- bytes/sec

它表示：

- 服务器每秒传输了多少字节的数据

这个指标在返回静态大文件时尤其重要。

## 7.3 成功请求数

表示：

- 服务器成功完成了多少请求

如果吞吐很高但成功率不高，那说明结果并不好。

## 7.4 失败请求数

表示：

- 建连失败
- 请求超时
- 收发异常
- 服务端崩溃或拒绝响应

这个指标对稳定性很重要。

## 7.5 并发客户端数

这个不是结果指标，而是实验条件。

比如：

- `-c 100`
- `-c 500`
- `-c 1000`

你要看的是：

- 随着并发增加，吞吐怎么变化
- 失败率是否上升
- 是否出现明显拐点

## 8. 项目里给出的压测结果说明了什么

README 里给了多组结果，对比的是：

- Reactor / Proactor
- LT / ET 的不同组合

它本质上在说明：

- 不同事件处理模型对吞吐有影响
- 不同触发模式组合对性能也有影响

这个很适合面试里说：

> 我不是只跑了一次压测，而是把服务器的不同并发模型配置拆开做了对比，这样能更清楚地看到系统设计对吞吐的影响。

## 9. 为什么不同配置下性能会不同

因为服务器的性能不是单一因素决定的，而是多模块共同作用的结果。

比如：

- LT 和 ET 模式对读写次数和事件触发频率有影响
- Reactor 和 Proactor 模式会影响主线程和工作线程的职责分配
- 日志开关会影响磁盘 I/O
- 线程池大小会影响任务并发处理能力
- 数据库连接池大小会影响数据库访问等待

所以压测本质上是在验证：

> 你的架构设计在真实高并发下表现怎样。

## 10. 如何根据压测结果分析性能瓶颈

这部分是面试最有价值的内容。

不要只会说：

- “QPS 是多少”

要会说：

- “QPS 为什么到这里上不去了”
- “失败率为什么升高”
- “瓶颈更像是 CPU、网络、磁盘，还是锁竞争”

## 10.1 如果并发增加，但 QPS 不再增长

可能说明：

- CPU 已经打满
- 线程池线程数不足
- 锁竞争严重
- 主线程 epoll 分发能力成为瓶颈

结合这个项目，常见怀疑点包括：

- 线程池任务处理跟不上
- `http_conn` 解析和响应构造耗时
- Reactor 模式下工作线程读写开销大

## 10.2 如果失败请求数开始明显增加

可能说明：

- fd 上限不够
- 监听队列太小
- 线程池积压严重
- 数据库连接池不够用
- 服务端响应不及时导致客户端超时

结合这个项目，还可能是：

- 定时器频繁关闭连接
- ET 模式读写处理不当
- 资源耗尽

## 10.3 如果吞吐高，但 bytes/sec 不高

说明：

- 请求可能都是小页面
- 服务器擅长处理轻量请求

这时候更像是在测：

- 调度能力
- 请求解析能力

而不是大文件传输能力。

## 10.4 如果 bytes/sec 很高，但 QPS 不一定高

说明：

- 请求可能是大文件
- 网络带宽或文件发送路径可能成为瓶颈

在这个项目里，这种情况下要重点怀疑：

- `mmap + writev` 路径表现
- 网络带宽
- 内核发送缓冲区

## 10.5 如果关闭日志后性能明显提升

说明：

- 日志 I/O 是明显瓶颈
- 磁盘写入拖慢了主流程

这正好可以说明：

- 为什么异步日志有意义

## 10.6 如果调大线程池后吞吐先升后降

说明：

- 线程数太少时并发不足
- 线程数太多时上下文切换和锁竞争增加

这意味着线程池不是越大越好。

## 10.7 如果涉及登录注册，请求一多就变慢

说明：

- 数据库连接池可能不够
- 数据库本身可能成为瓶颈
- 业务逻辑比静态文件请求更重

所以压测时最好区分：

- 静态资源测试
- 动态业务测试

## 11. 这个项目里最可能的瓶颈有哪些

按模块来分析，常见瓶颈可以这样拆。

## 11.1 主线程事件分发瓶颈

主线程要负责：

- `accept`
- `epoll_wait`
- 事件判断
- 分发任务

如果连接太多、事件太频繁，主线程可能先顶不住。

## 11.2 线程池处理瓶颈

如果工作线程数过少：

- 任务排队
- 响应延迟升高

如果工作线程过多：

- 上下文切换增多
- 锁竞争变多

## 11.3 锁竞争瓶颈

这个项目里多个地方都有锁：

- 线程池任务队列
- 日志系统
- 数据库连接池

高并发下锁竞争会拖慢吞吐。

## 11.4 数据库瓶颈

一旦请求进入登录注册流程，就不只是静态文件了。  
这时可能受限于：

- 数据库连接池大小
- MySQL 响应速度
- SQL 执行效率

## 11.5 网络和文件发送瓶颈

如果请求图片或视频文件，压力会更多落在：

- 网络带宽
- 文件发送路径
- 内核缓冲区

## 11.6 日志 I/O 瓶颈

如果开启同步日志：

- 每个请求都要写盘
- 很容易拖慢吞吐

## 12. 压测时应该怎么设计实验才更像工程分析

建议至少做下面几组对比。

### 对比 1：开日志和关日志

看日志系统是否成为瓶颈。

### 对比 2：Reactor 和 Proactor

看不同事件处理模型的吞吐差异。

### 对比 3：LT 和 ET

看不同触发模式组合的性能表现。

### 对比 4：不同线程数

比如：

- 4 线程
- 8 线程
- 16 线程

找最合适的线程池规模。

### 对比 5：静态请求和动态请求

把：

- 请求 html
- 请求登录注册

分开压测，区分网络瓶颈和数据库瓶颈。

## 13. 压测结果该怎么表达才像面试回答

不要只说：

> 我用 Webbench 测过，QPS 很高。

更好的说法是：

> 我用 Webbench 对服务器做过并发压测，主要从并发客户端数、压测持续时间、请求类型和服务器启动参数四个维度记录实验条件，再重点观察 QPS、字节吞吐和失败请求数。通过对比不同的 Reactor/Proactor、LT/ET 和日志开关配置，我能看到不同架构设计对吞吐的影响，并进一步分析是主线程分发、线程池调度、数据库访问还是日志 I/O 成为了性能瓶颈。

## 14. 你可以直接背的一段话

> Webbench 是一个轻量级 HTTP 压力测试工具，它通过模拟多个并发客户端在一定时间内持续向服务器发送请求，统计每秒请求数、数据吞吐量以及成功失败次数。我在这个项目里主要用它来对比不同并发模型和触发模式下的性能差异，比如 Reactor 和 Proactor、LT 和 ET 组合。压测时我会先固定请求资源和服务器参数，再观察 QPS、bytes/sec 和失败请求数这些指标。如果并发升高后 QPS 不再增长，通常说明 CPU、线程池或锁竞争到了瓶颈；如果失败数开始上升，通常说明 fd、线程池、数据库连接池或响应延迟出现了问题。这样压测的意义就不只是跑一个数字，而是帮助定位服务端的真实瓶颈。

## 15. 最后提醒

> [!warning]
> 压测结果一定不能脱离实验条件去讲。  
> 同样一个 QPS，换一个并发数、换一个请求文件、换一个日志开关，意义完全不同。

> [!tip]
> 面试里最加分的不是“你跑过压测”，而是“你知道为什么会跑出这个结果，以及下一步该优化哪里”。
