# TCP通信流程

## 它是什么

- TCP 通信流程指 TCP 客户端和服务器从创建 socket、建立连接，到收发数据、关闭连接的完整过程。

## 服务器端流程

- `socket`：创建监听 socket。
- `bind`：绑定服务器本地 IP 和端口。
- `listen`：进入监听状态。
- `accept`：等待客户端连接，成功后返回已连接 socket。
- `recv/read`：接收客户端数据。
- `send/write`：向客户端发送数据。
- `close`：关闭不再使用的 socket。

## 客户端流程

- `socket`：创建客户端 socket。
- `connect`：连接服务器 IP 和端口。
- `send/write`：发送数据。
- `recv/read`：接收服务器响应。
- `close`：关闭连接。

## 怎么理解

- 服务器的核心是“先等别人来连”。
- 客户端的核心是“主动连接服务器”。
- 连接建立后，双方就可以通过已连接 socket 双向通信。
- 建立连接不是应用层直接发送数据，而是先由内核完成 [[linux网络编程/概念词条/TCP三次握手|TCP三次握手]]。
- 关闭连接时，双方通常通过 [[linux网络编程/概念词条/TCP四次挥手|TCP四次挥手]] 释放两个方向的通信。
- 连接中的数据会被内核封装成 [[linux网络编程/概念词条/TCP数据包格式|TCP报文段]]，其中的序号、确认号和窗口大小支撑 [[linux网络编程/概念词条/TCP滑动窗口|TCP滑动窗口]]。
- 如果想把客户端代码、服务器代码、TCP 状态和报文方向放在一张图里看，见 [[linux网络编程/概念词条/TCP通信时序与代码对应关系|TCP通信时序与代码对应关系]]。
- 如果想从状态机角度看所有可能路径，见 [[linux网络编程/概念词条/TCP状态转换图|TCP状态转换图]]。

## 连接建立：三次握手

- 客户端调用 [[linux网络编程/函数笔记/Socket/connect|connect]] 主动发起连接。
- 服务器处于 `listen` 状态，内核处理握手队列。
- 握手完成后，服务器调用 [[linux网络编程/函数笔记/Socket/accept|accept]] 得到用于通信的 [[linux网络编程/概念词条/已连接套接字|已连接套接字]]。
- 详细过程见 [[linux网络编程/概念词条/TCP三次握手|TCP三次握手]]。

## 连接关闭：四次挥手

- 任意一方都可以主动关闭连接。
- 对端正常关闭后，本端 `recv/read` 通常返回 `0`。
- 关闭过程体现了 TCP 全双工通信的特点：两个方向可以分别关闭。
- 详细过程见 [[linux网络编程/概念词条/TCP四次挥手|TCP四次挥手]]。

## 数据传输：报文格式与滑动窗口

- 应用层调用 [[linux网络编程/函数笔记/Socket/send|send]] 后，内核会把数据组织成 [[linux网络编程/概念词条/TCP数据包格式|TCP报文段]]。
- TCP 首部中的序号、确认号和窗口大小用于描述“发到哪里、确认到哪里、还能接收多少”。
- [[linux网络编程/概念词条/TCP滑动窗口|TCP滑动窗口]] 允许发送方连续发送一段数据，而不是每发一个字节都等待确认。

## 易错点

- 服务器端 `accept` 返回的新 fd 才用于通信。
- 客户端 `connect` 传入的是服务器地址。
- 单连接阻塞式案例一次只能处理一个客户端。
- `recv/read` 返回 `0` 表示对端正常关闭，不是普通数据长度为 0。
