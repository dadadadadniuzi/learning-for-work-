# LungCancerScreening 口语化逐文件讲解稿

> [!summary]
> 这份笔记适合你“像给没接触过这个项目的人讲课一样”顺着讲。
> 建议你今晚按这里的顺序过一遍，明天面试时就不容易乱。

## 1. 我会怎么开场

如果让我从零讲这个项目，我会这样开头：

这个项目本质上是一个肺部 CT 的 AI 辅助筛查系统。你可以把它理解成一个三步走的流程。

第一步，不直接说这个病人有没有癌症，而是先在 CT 图像里找哪里看起来像结节。

第二步，对于这些可疑位置，再进一步判断它到底是不是真的结节。

第三步，如果确认是结节，再看它偏良性还是偏恶性。

这三步做完以后，再把结果通过后端接口和前端页面展示出来，让用户能够上传 CT、查看切片、看历史记录，甚至针对报告继续问 AI。

## 2. 我会怎么讲目录

我一般会把目录分成五层：

1. `util/` 是底层工具层。
2. `segment*` 是第一阶段，结节分割。
3. `dsets.py + model.py + training.py` 是第二阶段，真假结节分类。
4. `Tumor*` 是第三阶段，良恶性分类。
5. `web_app/` 是系统落地层，也就是后端和前端。

这样一讲，整个项目就不会显得散。

## 3. 先讲最底层：工具层

### 3.1 `util/util.py`

这个文件最重要的事情是做坐标转换。

因为医学影像里，标注点通常是病人坐标，比如 `x, y, z`，但代码裁图的时候要用数组索引，比如第几张切片、第几行、第几列。所以这个文件定义了两个核心函数：

1. `patientCoord2voxelCoord`
2. `voxelCoord2patientCoord`

我会把它讲成：

这个项目里最基础的一件事，就是把“医生世界里的坐标”翻译成“数组世界里的坐标”。如果这个转换错了，后面所有裁剪都会错。

这里还有一个 `enumerateWithEstimate`，它的作用不是算法，而是工程辅助。训练和缓存时，数据量非常大，这个函数能边跑边估算剩余时间，方便看进度。

### 3.2 `util/disk.py`

这个文件是缓存层。

我会这样讲：

CT 很大，如果每次训练都从完整 CT 里重新裁一个 3D 小块，速度会非常慢。所以作者用了 `diskcache` 做磁盘缓存，还加了 `gzip` 压缩。也就是说，第一次裁过的样本，以后可以直接从缓存里取。

这在医学影像项目里很常见，因为 IO 和预处理往往是瓶颈。

### 3.3 `util/unet.py`

这个文件是通用的 U-Net 实现，相当于项目里分割网络的底座。

我会这样讲：

作者没有从零手写一套特别复杂的分割网络，而是复用了一个比较标准的 U-Net 实现。这个 U-Net 有编码器、解码器和跳跃连接，适合保留医学图像里的空间细节。

## 4. 第一阶段：结节分割

### 4.1 先看 `segmentDsets.py`

这个文件是分割任务的数据来源。

我会这样讲：

分割任务的输入不是一个 3D 块，而是 2D 切片加上下文。项目会把一张切片前后各取几张，拼成多通道输入。比如这里默认前后各取 3 张，所以总共是 7 通道。

为什么这样做？因为结节是 3D 结构，只看单张切片，模型很容易把血管、噪点或者其他组织误判成结节。加上下文后，模型能更好理解这个东西在空间上是不是连续的。

这个文件里还有一个很关键的类是 `Ct`。它不只是读 CT，还会根据标注把结节区域做成 mask。也就是说，它一边提供原始图像，一边提供监督标签。

### 4.2 再看 `segmentModel.py`

这里的核心是 `UNetWrapper`。

我会这样讲：

`UNetWrapper` 是对底层 U-Net 的一个项目化封装。外面加了一层输入归一化，最后加了一个 `Sigmoid`，这样输出就变成 0 到 1 之间的分割概率图。

这个文件里还定义了 `SegmentationAugmentation`，它做的是 2D 数据增强。包括翻转、平移、缩放、旋转和加噪声。注意它不是只对图像增强，它会同时对 label mask 做同样的仿射变换，不然图和标签就对不上了。

### 4.3 再看 `segmentTraining.py`

这个文件是第一阶段训练主程序。

我会这样讲：

训练时，先构造训练 DataLoader 和验证 DataLoader，然后每个 batch 送入分割模型，算 Dice Loss。这里最值得说的一点是，作者对假阴性额外加了权重。换句话说，这个项目宁愿多报一点，也不想漏掉结节，因为在医疗筛查里漏诊往往比误报更危险。

这个脚本还做了两件比较工程化的事情：

1. 把指标和图像写进 TensorBoard
2. 每轮保存 checkpoint，并保存 best model

### 4.4 `prep_seg_cache.py`

这个文件不是训练本身，而是训练前的准备工作。

我会这样讲：

正式训练前，先让程序把分割数据集全过一遍，把常用样本缓存下来，后面训练就更快。

## 5. 第二阶段：真假结节分类

### 5.1 `dsets.py`

这个文件是第二阶段的数据集。

我会这样讲：

第一阶段输出的是一堆“看起来可疑”的区域，但这里面有很多不是真结节。所以第二阶段会以候选点为中心，从完整 CT 中裁一个固定大小的 3D 小块，比如 `32 x 48 x 48`，再送给分类模型。

这个文件里的 `getCandidateInfoList` 会把标注文件和候选文件合起来，得到一份候选列表。每个候选里有四个关键信息：

1. 是不是结节
2. 直径
3. 属于哪个 CT 序列
4. 中心坐标

然后 `LunaDataset` 会负责按索引把这些候选点真正裁成样本张量。

这里还有一个值得讲的点，就是它支持正负样本平衡采样。因为真实场景里负样本远多于正样本，如果不平衡采样，模型很容易学成“全都预测不是结节”。

### 5.2 `model.py`

这个文件定义了第二阶段的 3D CNN。

我会这样讲：

这里的模型整体结构比较简单，就是四个 3D 卷积块，每个块里是两层卷积加 ReLU，再做池化，最后拉平接全连接层，输出两类概率。

这个模型不是特别花哨，但对于固定大小的 3D 医学块分类任务来说，这种结构是很常见的。

### 5.3 `training.py`

这是第二阶段训练脚本。

我会这样讲：

它和分割训练很像，都是读数据、建模型、循环训练、验证、保存模型。不同点在于这里是分类任务，所以损失函数是交叉熵，指标也变成了 accuracy、precision、recall 和 F1。

我会特别强调一个点：

对于结节真假分类，F1 比 accuracy 更重要，因为数据不平衡。如果一个模型把所有候选都预测成负类，accuracy 可能看起来不差，但实际上没有筛出真正结节。

### 5.4 `prepcache.py`

作用和分割缓存脚本一样，就是训练前预热缓存。

## 6. 第三阶段：良恶性分类

### 6.1 `TumorDatasets.py`

这个文件的整体思路和 `dsets.py` 很像，但它进一步区分了：

1. 非结节
2. 良性结节
3. 恶性结节

我会这样讲：

这个阶段其实还是在裁 3D 小块，区别只是标签变了。前一阶段关注“有没有”，这一阶段关注“风险高不高”。

### 6.2 `TumorModel.py`

我会这样讲：

这个模型结构和真假结节分类模型很接近，本质上还是一个 3D CNN。这里多了一个 `augment3d`，说明训练时继续做了 3D 空间增强。

### 6.3 `TumorTraining.py`

这个文件特别适合拿来讲“迁移学习/微调”。

我会这样讲：

作者没有完全从头训练良恶性模型，而是从已有的结节分类模型出发，加载其参数，只解冻后面若干层来做 fine-tune。这样能利用前面已经学到的 3D 结构特征，同时降低小数据任务直接从零训练的不稳定性。

这里还会计算 AUC 和画 ROC 曲线，因为良恶性任务不能只看准确率，更要看模型在不同阈值下的区分能力。

### 6.4 `TumorPrepCache.py`

还是同样的角色，做缓存预热。

## 7. 把三阶段串起来：`model_evel.py`

这个文件非常关键，因为它告诉我们这个项目不是孤立训练三个模型，而是把它们串成了完整系统。

我会这样讲：

这份脚本做的事情是：

1. 加载分割模型
2. 加载真假结节分类模型
3. 加载良恶性分类模型
4. 对某个 CT 或整个验证集逐个处理
5. 把三个阶段串起来做推理
6. 最后和真实标注做匹配，输出混淆矩阵

这里的 `match_and_score` 很值得讲，因为它不是简单按类别比较，而是先看预测点和真实结节中心是否足够接近，距离在结节直径范围内才算匹配成功。这个思路很符合目标检测或医学定位任务。

如果面试官问“你怎么看这个评估方式”，你可以说：

我觉得这个评估方式比单独看分类分数更真实，因为它评估的是整条 pipeline，包含候选生成、定位误差和后续分类，而不是只看一个干净数据集上的分类结果。

## 8. 后端：`web_app/app.py`

这是系统落地的核心。

我会这样讲：

前面的代码都还是偏研究型代码，到这里开始变成产品代码了。`app.py` 里有两个关键部分：

1. `Diagnosis` 数据库模型，负责存历史诊断记录
2. `TumorPredictionSystem`，负责封装完整推理流程

### 8.1 `TumorPredictionSystem`

这个类的逻辑很好讲，因为它几乎就是整个产品的核心业务流程。

我会这样说：

当用户上传一组 CT 文件后，系统先在临时目录里把文件落盘，然后读取 CT 数据，接着调用 `_segment_ct` 做逐切片分割，再通过 `_group_segmentation_output` 做连通域分析，把可疑区域转成候选点。然后系统再调用 `_classify_candidates`，把每个候选块送到真假结节分类模型和良恶性模型里，最后把所有结果汇总成前端能展示的结构化 JSON。

### 8.2 API 层

我会把这些接口概括成三类：

1. 诊断接口：`/api/upload`
2. 展示接口：`/api/predictions`、`/api/statistics`、`/api/diagnosis-*`
3. 交互接口：`/api/chat`、`/api/ct-slice/...`

如果时间很紧，我只重点讲两个：

#### `/api/upload`

这是主业务接口。

用户上传 `.mhd` 和 `.raw` 后，后端完成完整推理，然后把结果存数据库并返回。

#### `/api/ct-slice`

这是可视化接口。

前端切换切片时会请求这个接口，后端会动态生成一张 PNG，并把对应结节画框，这样用户就能在页面里像看片一样查看 CT。

### 8.3 `/api/chat`

这个接口很像“锦上添花”的功能。

我会这样讲：

作者把大模型问答接进来了。聊天接口支持带 `predictionId`，这样用户不是泛泛地问，而是可以围绕某一份具体报告继续提问。后端会先从数据库里取出那份报告的摘要，再把报告上下文和用户问题拼到 prompt 里发给通义千问。

这体现出项目不只是做“模型推理”，还想往“智能报告交互”方向扩展。

## 9. 一键启动：`web_app/start_system.py`

这个文件我会简短讲：

它是个工程脚本，负责检查 Node.js 和 npm 是否安装、必要时安装前端依赖，然后同时启动 Flask 后端和 React 前端。说明作者有考虑别人怎么快速跑起整个系统，而不是只关心模型代码。

## 10. 前端部分怎么讲

前端我不会逐行讲，而是按页面职责讲。

### 10.1 `App.js`

作用是路由入口。

### 10.2 `Layout.js`

作用是整体界面骨架，负责菜单和内容区。

### 10.3 `Upload.js`

这是最重要的页面，因为它直接承接核心业务。

我会这样讲：

上传页让用户分别上传 `.mhd` 和 `.raw`，前端先检查两个文件是不是同一组 CT，然后把它们发到后端。返回结果后，页面会展示总体结论、最可疑结节、结节列表，还可以打开切片浏览器。

### 10.4 `CTViewer.js`

这是可视化亮点。

我会这样讲：

这个组件会自动定位到最可疑结节所在的切片，并动态请求后端生成带框的 CT 图像，用户还能拖动滑块看前后切片。这个功能让整个项目更像一个真实医疗辅助系统，而不是一堆数字结果。

### 10.5 `Dashboard.js`

这是仪表盘，用来展示系统状态、统计信息和图表。

### 10.6 `History.js`

这是历史报告中心。

可以看历史记录、展开详情、查看 CT、并跳到聊天页继续讨论同一份报告。

### 10.7 `ModelStatus.js`

这个页面告诉用户当前模型是否加载成功、是不是运行在真实预测模式。

### 10.8 `Chat.js`

这个页面承接 AI 问答。

如果用户是从历史记录点进来的，还会自动带上那份报告的上下文。

## 11. 如果面试官让我讲“项目实现逻辑”

我会用下面这段：

这个项目的实现逻辑其实很清晰。底层先解决数据问题，也就是如何从 CT 文件里读出体数据、如何在不同坐标系之间转换、如何裁出模型需要的样本块。然后第一阶段用分割模型提高召回，把可疑区域尽量找全；第二阶段用 3D 分类器降低误报，只保留真正像结节的候选块；第三阶段在真实结节上做良恶性预测。最后把这一整条推理链路封装进 Flask 服务，再由 React 前端做可视化交互。

## 12. 如果面试官让我讲“你从这个项目里学到了什么”

我会这样说：

我最大的收获是理解了一个医学 AI 项目从研究代码到系统原型之间的过渡过程。不是说训练出一个模型就结束了，真正要落地还要考虑数据读取、缓存、推理链路组织、评估方式、接口设计、前端展示和用户交互。我也更清楚地理解了为什么很多医疗 AI 系统不会直接做单模型端到端，而是采用多阶段流水线来提升可解释性和可控性。

## 13. 如果面试官问“你觉得这个项目哪里还能改”

我会这样讲：

如果继续做，我会优先改三类问题。

第一类是工程问题，比如数据库配置和模型路径现在有硬编码，应该配置化。

第二类是代码质量问题，比如个别文件里有历史遗留逻辑和命名不统一的问题。

第三类是产品化问题，比如统计页里有一些模拟数据，后续应该接真实监控。

如果再往模型方向走，也可以继续优化候选后处理、3D backbone、类别不平衡策略和更严格的医学评估标准。

## 14. 最适合你明天现场讲的版本

如果面试官说“你顺着讲讲这个项目”，你可以基本按下面顺序：

1. 先说这是一个三阶段肺癌筛查系统。
2. 再说第一阶段分割找候选区域。
3. 再说第二阶段 3D CNN 判断真假结节。
4. 再说第三阶段做良恶性分析。
5. 再说项目还做了缓存、评估脚本、Flask 后端和 React 前端。
6. 最后说优点和可改进点。
