# 半同步半反应堆与并发模型详解

返回：[[TinyWebServer-面试拆解笔记]]

相关：[[TinyWebServer-拆解/03-入口与WebServer总控]]、[[TinyWebServer-拆解/05-线程池与并发模型]]、[[TinyWebServer-面试问答模拟版]]

> [!abstract]
> 这篇专门讲清楚：
> 1. 什么是“半同步 / 半反应堆”
> 2. 它在这个项目里是怎么实现的
> 3. 工作线程为什么叫“同步处理任务”
> 4. 同步和异步到底区别在哪
> 5. Reactor、Proactor、主从 Reactor 的区别

## 1. 先讲一句最适合面试的定义

> “半同步 / 半反应堆”模型的核心思想是：主线程只负责监听 I/O 事件和分发任务，工作线程同步处理具体请求逻辑。也就是把‘事件就绪检测’和‘任务实际处理’拆开。

这句话你可以直接背。

## 2. 什么叫“反应堆”

“反应堆”对应的是 `Reactor` 思想。  
它的重点不是“处理业务”，而是：

- 监听 I/O 事件
- 发现哪个 fd 就绪了
- 决定接下来该把谁交给谁处理

换句话说，反应堆层最关心的是：

- 哪个 socket 可读
- 哪个 socket 可写
- 哪个 socket 断开了
- 哪个新连接到来了

在这个项目里，这一层主要由：

- `epoll`
- 主线程事件循环
- `WebServer::eventLoop()`

来承担。

## 3. 什么叫“半反应堆”

“半反应堆”里的“半”，指的是：

- 主线程只负责一部分反应堆职责
- 只做事件监听和事件分发
- 不把所有逻辑都塞进主线程

也就是说，它不是那种：

- 主线程既监听事件
- 又自己把业务逻辑全做完

而是监听完之后，把真正的处理交给别的线程。

所以在这个项目里：

- 主线程是“事件分发者”
- 工作线程是“任务执行者”

这就是“半反应堆”的含义。

## 4. 什么叫“同步”

这里的“同步”不是说“只有一个线程”，也不是说“没有并发”。  
这里的“同步”是指：

> 工作线程一旦拿到任务，就按顺序把这个任务处理完，中间不会把这个任务再拆成更小的异步回调流程。

例如在这个项目里，一个工作线程拿到某个连接任务后，会同步地做这些事：

- 读数据或写数据
- 解析 HTTP 报文
- 访问数据库
- 构造响应

这些步骤是在这个线程里连续执行完成的。

## 5. 什么叫“半同步”

“半同步”里的“半”指的是：

- 任务处理层由线程池同步执行
- 但事件监听层不是同步阻塞地一个一个处理，而是由反应堆模式统一分发

所以整个系统是两层结构：

### 第一层：异步感知事件

- 谁可读
- 谁可写
- 谁超时
- 谁断开

### 第二层：同步执行任务

- 工作线程取任务
- 一个任务一个任务处理完

所以它叫：

- 半同步
- 半反应堆

## 6. 这个项目里“半同步 / 半反应堆”是怎么实现的

## 6.1 主线程负责什么

主线程主要在 `WebServer::eventLoop()` 里工作。

它负责：

- `epoll_wait()` 等待事件
- 判断是监听 socket 事件，还是连接 socket 事件
- 如果是新连接，就 `accept`
- 如果是读写事件，就把任务交给线程池
- 如果是信号事件，就处理定时器或退出逻辑

也就是说，主线程并不专门去执行业务逻辑，而是做：

- 监听
- 判断
- 分发

## 6.2 工作线程负责什么

工作线程主要在 `threadpool::run()` 里工作。

它会：

1. 等待任务队列中有任务
2. 被唤醒
3. 取出一个 `http_conn` 请求对象
4. 对这个连接执行对应的读、写或处理逻辑
5. 处理完后再回到等待状态

这就是典型的同步任务处理。

## 6.3 这两层是怎么连接起来的

连接点就是：

- 主线程把任务放进线程池队列
- 工作线程从队列中取任务执行

在代码里体现为：

- 主线程调用 `m_pool->append(...)` 或 `m_pool->append_p(...)`
- 工作线程在 `run()` 中从 `m_workqueue` 取任务

## 7. 为什么这种模型适合服务器

因为服务器里最常见的问题是：

- 同时连接很多
- 但不是所有连接都一直活跃

如果不用这种模型，而是：

- 一个连接一个线程

就会带来：

- 线程太多
- 上下文切换多
- 资源浪费严重

如果用纯单线程：

- 又容易被慢请求拖住

而“半同步 / 半反应堆”正好兼顾了：

- 事件分发效率
- 任务处理并发能力

## 8. 工作线程为什么叫“同步处理任务”

这个问题面试很爱追问。

你可以这样理解：

### 同步处理的意思

线程拿到一个任务后：

- 直接处理到底
- 在当前线程里一步一步做完
- 当前步骤不完成，后一步就不会开始

例如这个项目里，一个线程拿到请求后，可能这样执行：

1. `read_once()`
2. `process_read()`
3. `do_request()`
4. `process_write()`
5. `write()`

这是一个顺序执行流程。

### 它为什么不是异步处理

因为这里不是：

- 发起一个处理请求
- 立刻返回
- 然后靠回调通知后续逻辑

也不是：

- 一个任务拆成很多 future/promise 链式执行

它就是典型的：

- 线程取任务
- 线程自己把任务做完

所以叫同步处理。

## 9. 同步和异步到底怎么区分

这是最容易说乱的地方。

最稳妥的理解是：

### 同步

> 调用者必须等待这个操作完成，或者在当前控制流里顺序推进。

例如：

- 当前线程调用函数
- 这个函数没做完，当前流程就不会往后走

### 异步

> 调用者发起一个操作后可以先继续做别的事，真正完成时再通过通知、回调或事件机制告知结果。

例如：

- 发起 I/O
- 当前线程先返回
- 以后再通知“完成了”

## 10. 在这个项目里，哪些部分更像同步，哪些更像异步

### 更像异步的部分

- `epoll_wait()` 监听事件
- 信号转管道统一处理
- 主线程“收到事件再分发”

这些都是事件驱动风格。

### 更像同步的部分

- 工作线程拿到任务后的处理流程
- HTTP 状态机解析
- 注册登录逻辑
- 响应构造

这些都是在一个线程里顺序完成的。

## 11. Reactor 是什么

Reactor 的核心思想是：

> 内核只告诉你“某个 fd 已经就绪”，真正的读写由应用程序自己做。

所以 Reactor 模式里：

- `epoll` 告诉你哪个 fd 可读
- 你再调用 `recv()`
- `epoll` 告诉你哪个 fd 可写
- 你再调用 `send()` 或 `write()`

### 这个项目里的 Reactor 怎么体现

在这个项目里，`actor_model = 1` 时走 Reactor 风格。

主要特点是：

- 主线程检测到读事件
- 把“读任务”交给线程池
- 工作线程去真正执行 `read_once()`

写事件同理：

- 主线程把“写任务”交给线程池
- 工作线程去执行 `write()`

## 12. Proactor 是什么

严格意义上的 Proactor 核心思想是：

> 应用发起异步 I/O 请求，真正的 I/O 由内核完成，完成后再通知应用程序。

也就是说：

- Reactor 是“你自己做读写”
- Proactor 是“内核帮你做完读写，再告诉你结果”

### 这个项目里的 Proactor 为什么是“工程近似版”

这个项目里的 `actor_model = 0` 被称为 Proactor，但更准确地说它是：

- 由主线程先完成读
- 再把逻辑处理交给线程池

也就是：

- 主线程先 `read_once()`
- 读完后把任务交给线程池
- 工作线程主要负责 `process()`

所以它不是严格内核 AIO 意义上的 Proactor，而是工程实现里常见的“主线程先完成 I/O，再分发处理”的近似 Proactor 模式。

## 13. Reactor 和 Proactor 的区别怎么讲最稳

可以直接背这段：

> Reactor 模式下，epoll 只负责通知某个 fd 已经可读可写，真正的读写动作由应用线程完成；Proactor 模式下，I/O 会先完成，再把完成后的结果交给后续逻辑处理。在这个项目里，Reactor 是把读写任务交给线程池线程处理，而所谓 Proactor 是主线程先把数据读好，再交给线程池做业务处理。

## 14. 主从 Reactor 是什么

主从 Reactor 可以理解为对 Reactor 的进一步扩展。

### 主 Reactor

负责：

- 监听监听 socket
- 接收新连接

### 从 Reactor

负责：

- 监听已经建立连接的 socket
- 处理这些连接上的读写事件

这种模式适合：

- 并发量更高
- 连接数更多
- 单个主线程仅做接入分发

## 15. 主从 Reactor 和这个项目的关系

这个项目没有做到严格意义上的“主 Reactor + 多个从 Reactor”那种分层模型。  
它更接近于：

- 一个主线程负责统一监听和分发
- 一组工作线程负责同步处理任务

所以你可以说：

> 这个项目不是严格的主从 Reactor，而是更典型的半同步 / 半反应堆模型。

## 16. 这三者之间怎么快速对比

### Reactor

- 关注点：事件就绪通知
- 读写动作：应用自己做

### Proactor

- 关注点：I/O 完成通知
- 读写动作：理想情况下内核完成

### 主从 Reactor

- 关注点：把 Reactor 再拆层
- 主 Reactor 管连接建立
- 从 Reactor 管连接 I/O

## 17. 为什么这个项目要支持 Reactor 和 Proactor 两种模式

因为这样可以：

- 对比不同 I/O 分工方式的效果
- 展示作者对并发模型的理解
- 在压测时比较不同模型的性能差异

面试里这也很加分，因为说明你不是只会“写个能跑的服务器”，而是知道：

- 主线程该干什么
- 工作线程该干什么
- 模型不同，性能表现也可能不同

## 18. 这个模型的优点是什么

- 主线程轻量，接入能力强
- 工作线程专注处理任务，职责清晰
- 比一连接一线程更节省资源
- 比纯单线程更能利用多核
- 容易和线程池、定时器、日志系统结合

## 19. 这个模型的局限是什么

- 主线程仍可能成为分发瓶颈
- 工作线程同步处理重任务时会拖慢后续任务
- 线程池大小不好设，太小吞吐不足，太大竞争严重
- 不如更复杂的主从 Reactor 那么容易横向扩展

## 20. 面试里最推荐的一段回答

> “半同步 / 半反应堆”模型可以理解成两层：第一层是反应堆层，主线程用 `epoll` 统一监听事件，只负责发现谁可读、谁可写、谁有新连接，然后把任务分发出去；第二层是同步处理层，工作线程从线程池中取任务后，在当前线程里按顺序把这个请求处理完，比如读数据、解析 HTTP、访问数据库、构造响应。这个项目就是这样实现的，所以主线程更轻量，工作线程负责真正业务处理。它整体上不是严格的主从 Reactor，而是更典型的半同步 / 半反应堆模型。同时项目还支持 Reactor 和 Proactor 两种风格：Reactor 下工作线程做真正的读写，Proactor 下主线程先完成读，再把任务交给线程池做逻辑处理。”

## 21. 你明天最少要记住的 6 句话

1. 半反应堆指主线程只负责事件监听和分发，不负责全部业务。
2. 半同步指工作线程从线程池取任务后，顺序把任务处理完。
3. 这个项目里主线程核心是 `epoll_wait + 事件分发`。
4. 工作线程核心是 `线程池取任务 + http_conn 处理请求`。
5. Reactor 是“事件就绪后应用自己读写”，Proactor 是“I/O 完成后再处理逻辑”。
6. 这个项目整体更像半同步 / 半反应堆，而不是严格主从 Reactor。
