# Workflow-09 高频面试题与背诵稿

相关笔记：
[[Workflow项目面试拆解]] | [[Workflow-01 项目定位与一句话介绍]] | [[Workflow-04 HTTP 请求与服务端链路]] | [[Workflow-08 路由、DNS、协议层]]

## 1. 这个项目是什么

推荐回答：

这个项目不是业务系统，而是一个 C++ 异步网络工作流框架。它把网络请求、文件 IO、定时器、计算任务统一抽象成 Task，再通过串行流和并行流组织成 Workflow。底层是异步通信和调度引擎，上层提供 HTTP、Redis、MySQL、DNS 等任务和服务端封装。

## 2. 它解决了什么问题

推荐回答：

它主要解决的是 C++ 网络开发中底层复杂度太高的问题，比如 socket、epoll、线程池、协议解析、超时控制、回调嵌套和多任务编排。这个框架把这些通用复杂度封装起来，让业务层只需要描述任务和任务之间的关系。

## 3. 为什么说它不是普通线程池项目

推荐回答：

普通线程池项目主要解决的是计算任务调度，而这个项目还统一处理了网络 IO、服务端监听、协议解析、DNS、路由和服务治理，所以它不只是线程池，而是完整的异步网络框架。

## 4. 这个项目为什么快

推荐回答：

它快的核心原因是异步非阻塞模型，不会一请求一线程；另外它统一了网络 IO、定时器、文件 IO 和线程池任务的调度，并且支持连接复用和 keep-alive，所以吞吐会比较高。

## 5. 它最核心的抽象是什么

推荐回答：

我认为最核心的抽象是 `SubTask`、`SeriesWork` 和 `ParallelWork`。前者定义了统一任务模型，后两者定义了任务之间的顺序关系和并发关系。

## 6. 一个 HTTP 请求在里面怎么走

推荐回答：

用户先通过 `WFTaskFactory` 创建 HTTP 任务，任务启动后进入 `CommScheduler`，调度器选择目标和连接，再由 `Communicator` 完成 connect、send、recv，之后协议层把字节流解析成 `HttpResponse`，最后触发 callback。如果任务属于某个 `SeriesWork`，还会继续调度后续任务。

## 7. 一个 HTTP Server 请求怎么走

推荐回答：

`WFHttpServer` 启动监听后，收到新连接会创建会话和任务对象，协议层先解析请求，再执行用户的 `process(task)` 回调，用户在回调里填充响应对象，框架最后负责把响应发回去，并决定是否 keep-alive。

## 8. 如果你从零实现，会怎么做

推荐回答：

我会先定义统一任务抽象，再实现串行和并行工作流，然后做底层异步通信和调度器，再把网络、定时器、文件 IO、计算任务都包装成 Task，最后补协议层、Server/Client、DNS、路由和 upstream 等能力。

## 9. 它和 Boost.Asio 的区别怎么说

推荐回答：

两者都能做异步网络编程，但 Asio 更偏底层通用 IO 工具箱，而 Workflow 更强调高层任务封装、工作流编排和内置协议能力，所以更接近业务开发框架。

## 10. 最后可以直接背的总结

这个项目是一个 C++ 异步网络工作流框架。它的核心不是某一个协议，而是统一任务模型、统一调度模型和统一编排模型。底层通过 `Communicator` 和 `CommScheduler` 提供异步通信和连接管理，上层通过 `WFTaskFactory`、Server、Client、DNS、路由和协议模块把这些能力包装成开发者可直接使用的接口。
