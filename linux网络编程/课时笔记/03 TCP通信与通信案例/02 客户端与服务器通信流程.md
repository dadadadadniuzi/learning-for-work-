## 本节学什么详解

- 服务器流程：服务器先调用 `socket` 创建监听 socket，再通过 `bind` 绑定本地 IP 和端口，通过 `listen` 进入监听状态，最后调用 `accept` 得到用于通信的已连接 socket。
- 客户端流程：客户端调用 `socket` 创建 socket，准备服务器地址结构，然后调用 `connect` 主动发起连接，连接成功后进入读写阶段。
- [[linux网络编程/概念词条/TCP三次握手|TCP三次握手]]：`connect` 发起连接请求，服务器监听队列接收连接请求，双方通过 `SYN`、`SYN+ACK`、`ACK` 建立连接。
- [[linux网络编程/概念词条/TCP四次挥手|TCP四次挥手]]：关闭连接时，主动关闭方和被动关闭方分别关闭发送方向，应用层常见现象是 [[linux网络编程/函数笔记/Socket/recv|recv]] / [[linux网络编程/函数笔记/Socket/read|read]] 返回 `0`。
- [[linux网络编程/概念词条/TCP半关闭|TCP半关闭]]：只关闭一个方向的通信，例如 `shutdown(fd, SHUT_WR)` 表示本端不再发送，但仍可继续接收。
- [[linux网络编程/概念词条/TCP数据包格式|TCP数据包格式]]：连接建立、数据确认、连接关闭都依赖 TCP 首部字段，比如序号、确认号、窗口大小和标志位。
- [[linux网络编程/概念词条/TCP滑动窗口|TCP滑动窗口]]：数据传输阶段，发送方根据接收窗口和确认号决定还能连续发送多少数据。
- [[linux网络编程/概念词条/TCP通信时序与代码对应关系|TCP通信时序与代码对应关系]]：用一张时序图把 `connect`、`accept`、`read/write`、`close` 与 `SYN`、`ACK`、`DATA`、`FIN` 对起来。
- [[linux网络编程/概念词条/TCP状态转换图|TCP状态转换图]]：用状态机视角看客户端、服务器在建立连接和关闭连接时分别经过哪些 TCP 状态。
- [[linux网络编程/概念词条/阻塞式IO|阻塞式通信流程]]：在默认阻塞模式下，`accept` 会等待客户端连接，[[linux网络编程/函数笔记/Socket/recv|recv]] / [[linux网络编程/函数笔记/Socket/read|read]] 会等待数据到达，`send/write` 在发送缓冲区不可用时也可能阻塞。
- 连接关闭时读写函数的表现：对端正常关闭连接后，本端 [[linux网络编程/函数笔记/Socket/recv|recv]] / [[linux网络编程/函数笔记/Socket/read|read]] 通常返回 `0`；出错时返回 `-1` 并设置错误信息。

## 知识点补充

- 服务器端 `bind` 的地址是服务器自己的地址，客户端 `connect` 的地址是服务器地址。
- `accept` 带出的客户端地址可以用 [[linux网络编程/函数笔记/地址转换/inet_ntop|inet_ntop]] 和 [[linux网络编程/函数笔记/网络字节序/ntohs|ntohs]] 打印。
- TCP 是双向通信，连接建立后客户端和服务器都可以读也可以写。
- TCP 建立连接对应 [[linux网络编程/概念词条/TCP三次握手|TCP三次握手]]，释放连接对应 [[linux网络编程/概念词条/TCP四次挥手|TCP四次挥手]]。
- 半关闭体现了 TCP 的双向通信特点：本端可以关闭写方向，同时继续读对端数据，见 [[linux网络编程/概念词条/TCP半关闭|TCP半关闭]]。
- TCP 的可靠传输不是 socket 函数本身保证的，而是由 [[linux网络编程/概念词条/TCP数据包格式|TCP数据包格式]]、确认机制、重传机制和 [[linux网络编程/概念词条/TCP滑动窗口|TCP滑动窗口]] 共同支撑。
- 单连接案例的限制是一次只能服务一个客户端；后面多进程、多线程和 IO 多路复用就是为了解决并发连接处理问题。

## 本节内容速览

- 服务器负责监听和接收连接。
- 客户端负责主动连接。
- `accept` 成功后，双方使用已连接 socket 通信。
- 连接关闭、返回值和错误处理是 TCP 代码必须检查的重点。
